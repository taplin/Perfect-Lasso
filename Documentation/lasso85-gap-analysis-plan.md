# Lasso 8.5 Language Compliance Gap Analysis — Top-Down Pass

## Methodology note

This review inventoried the Lasso 8.5 Language Guide (849 pages) chapter-by-chapter — String Operations (25), Regular Expressions (26), Bytes (27), Math (28), Date/Time (29), Arrays/Maps/Compound Types (30), Files (31), Variables (15), Conditional Logic (16), Control Tags (43), HTTP/HTML Content (49), plus TOC-level inventory of every remaining chapter (Images 32, Networking 33, XML 34, PDF 35, JavaBeans 36, iCalendar 37, Process/Shell 38, LDAP 39, Namespaces 40, Logging 41, Encryption 42, Threads 44, Compound Expressions 45, Misc 46, Email/POP 47-48, XML-RPC 50, SOAP 51, Wireless 52, AJAX/LJAX 53, DNS 54, Custom Tags 57, Custom Types 58, Custom Data Sources 59) — and cross-referenced every documented tag/member/symbol against the actual registries in `Sources/LassoParser/`: `TagCatalog.swift` (block-tag parsing only), `Runtime.swift`'s `registerDefaultFunctions()` (the complete list of ~64 free native functions — verified exhaustively via grep, not sampled), `NativeTypes.swift`'s `LassoNativeTypeRegistry` (only `web_request`, `web_response`, `date`, `bytes` exist as native types), and `Evaluator.swift`'s `member(_:_:_:)` switch (the complete string/array/map/pair member-dispatch surface, also verified exhaustively).

Items already tracked in `Documentation/compatibility-matrix.md`, `outstanding-compatibility-project-plans.md`, `legacy-define-tag-type-plan.md`, and `library-and-custom-tags.md` are mentioned only in one line each, not re-analyzed — those documents record real, already-decided corpus-driven work (sessions, inline/database actions, FileMaker, web_response includes, `Date_Format`, `Decode_Base64`, `Encrypt_HMAC`, `Currency`/`Percent`, `Select`/`Case`, container tags/`-Priority`/`-Criteria` deferrals, legacy `define_tag`/`define_type` flag classification). Everything below is what a corpus-only approach could not have found because the corpus never exercised it.

## Implementation status (2026-07-18 — branch `language-primitives-p0-batch`)

The overlapping P0 items across this doc and `lasso9-lassoguide-gap-analysis-plan.md` were implemented, each reviewed by an application architect, a Swift concurrency reviewer, and a standard code reviewer before landing:

- **✅ DONE — `Loop_Abort`/`Loop_Continue`** (Section 1) — real break/continue for `loop`/`while`/`iterate`/`records`/`with`, correctly scoped to the nearest enclosing loop (a loop nested inside another loop, or a custom tag call with no loop of its own, doesn't leak the signal outward). `loop_key()` also added. Two real bugs found and fixed during review: a stray call with no enclosing loop was truncating the rest of the page (should be a no-op), and a signal set inside a called custom tag could leak out and hijack the caller's own loop.
- **✅ DONE — Array member-tag expansion** (Section 4) — `->Last`/`->Second`/`->Reverse`/`->Sort`/`->Join`/`->Contains`/`->Find`/`->FindPosition`/`->Remove`/`->RemoveAll`. Verified directly against the PDF's own worked examples; an initial draft had `->Find`/`->FindPosition` conflated with `->Contains` and `->Remove`/`->RemoveAll`'s documented semantics swapped, both caught by architect review re-reading the source pages.
- **✅ DONE — Map member-tag expansion** (not this doc's own finding, see the LassoGuide 9 doc's Section 2) — `->Size`/`->Keys`/`->Values`/`->Contains`/`->Find`/`->Remove`/`->RemoveAll`, plus a fix for the real `Map->Size` name-collision bug this doc predicted.
- **✅ DONE (partial) — `Null->Type`/`Var_Reset`/references** (Section 5, `language-primitives-batch2` branch) — `Null->Type`/`->IsA`/`->IsNotA`/`->HasMethod` implemented and verified against Ch. 43 Table 6 (see the LassoGuide 9 doc's Section 10 status entry for the full writeup). `Var_Reset`/`Local_Reset`/`Global`/`Global_Reset`/`Global_Defined`/`Global_Remove`/`Globals` also implemented, backed by a genuinely separate true-global storage namespace so page `Variable`s and true `Global`s sharing a name stay independent. This section's own predicted risk was directly verified, not assumed: confirmed by grep that this codebase's parameter binding and `Iterate` loop-variable binding do NOT implement real reference semantics at all — there is no `@`-operator parser support anywhere and variable storage is plain, unboxed values, so any existing "mutation propagates to the caller" behavior is Swift class-reference semantics on `.object` values specifically, not a general reference mechanism, and the documented `@`/`[Reference]` variable-aliasing system (Table 6) — including the `Ex_Square` in-place-array-mutation pattern this section cites — remains genuinely unimplemented, deferred as a separately-scoped follow-up needing new parser support plus a storage-indirection layer. `Var_Reset`/`Local_Reset`/`Global_Reset` are implemented as plain synonyms for their base tag accordingly (nothing to detach). True cross-request Global persistence (vs. this implementation's per-page-render scope) is also out of scope — see the LassoGuide 9 doc's status entry for the full disclosure.
- **Partially addressed — Math_\* substitution-tag family** (Section 6) — `Math_Abs`/`Add`/`Sub`/`Mult`/`Div`/`Mod`/`Max`/`Min`/`Ceil`/`Floor`/`RInt`/`Round`/`Random`/`Sqrt`/`Pow` implemented and verified against the Guide's own worked examples (one confirmed real documentation defect found in `Math_Div`'s own page-370 examples — implemented against the clean, internally-consistent rule instead, not the outlier numbers). `Math_ConvertEuro`/`Math_Roman`, the full trig set beyond `Sqrt`/`Pow`, `Integer->SetFormat`/bit-operations, and `Decimal->SetFormat` remain open.
- **✅ DONE (partial) — Date/Time** (Section 7) — `Date_Add`/`Date_Subtract`/`Date->Add`/`Date->Subtract` plus field accessors (`->year`/`->month`/`->day`/`->dayOfMonth`/`->hour`/`->minute`/`->second`/`->dayOfWeek`/`->asInteger`) that this doc didn't call out but were an equally foundational gap (see the LassoGuide 9 doc's Section 4). A real aliasing bug was found and fixed: an initial `->Add`/`->Subtract` implementation mutated the date object's own class instance in place, which silently corrupted any OTHER variable that happened to reference the same object after a plain assignment (`var(d2 = $d1)`) — fixed to match real Lasso's documented copy-on-assignment semantics.

Everything else below (Regular Expressions, the full Array/Map/Compound-type subsystem beyond the members above, Files, Variables/References, Encryption/Serialization/Compression, Web/HTTP infrastructure beyond what's tracked, Namespaces/DNS/Threads/Unique ID, Email/XML/lower-priority chapters) is still open, unchanged from the original analysis below.

## 1. Control Flow (P0 territory — core language primitives)

**`Loop_Abort` / `Loop_Continue`** — Priority: **P0**. Status: missing entirely. Confirmed by exhaustive grep: `loop_count` is implemented (`Renderer.swift:165,300`, set as a local var during `loop`/`iterate`/`records` rendering) but there is no `Loop_Abort` or `Loop_Continue` native anywhere in the codebase, and no early-exit/skip signal in the loop-rendering machinery at all. This is Lasso's `break`/`continue` — documented in Chapter 16 as usable inside *any* looping container (`Loop`, `Records`, `While`, `Iterate`). A tree-walking loop with no way to exit early is a genuine correctness gap, not an edge case — pagination "stop after N matches" and search-cutoff patterns depend on it. Rationale: fundamental control-flow primitive absent from an otherwise-complete `if`/`loop`/`while`/`iterate` implementation. Suggested location: `Renderer.swift` (needs a new signal alongside the existing `returnSignal` mechanism — likely `loopAbortSignal`/`loopContinueSignal` on `LassoContext`, checked in each of `loop`/`while`/`iterate`/`records`'s render loops) plus two registrations in `Runtime.swift`.

**`If_True`/`If_False`/`If_Empty`/`If_Null`** (Ch 16, Table 4) — Priority: **P1**. Status: missing entirely (verified: no `if_true`/`if_false`/`if_empty`/`if_null` registrations). These are expression-form conditionals commonly used inline (`[Var: 'x' = (If_Empty: $y, 'default')]`) as an alternative to the `?|` ternary (which *is* implemented). Suggested location: `Runtime.swift`, alongside the existing ternary-operator handling in `ExpressionParser.swift`.

**`with x in y do { }`** and colon-call block openers — already implemented per `library-and-custom-tags.md`; not re-analyzed.

## 2. String Handling

Current surface (`Evaluator.swift` member dispatch): `size`, `uppercase`, `lowercase`, `asstring`, `encodehtml`, `encodeurl`, `encodesmart`, `encodebreak`, `encodexml`, `encodestricturl`, `encodesql`, `encodebase64`, `decodebase64`, `contains`, `split`, `replace`, `append`, `trim`, `substring`/`sub`. That is roughly 19 of the ~50 documented string members (Ch 25, Tables 3, 5, 7, 9, 11, 13) plus none of the free-tag "substitution" family (Ch 25, Tables 4, 6, 8, 10).

- **`String->BeginsWith`/`->EndsWith`/`->Find`/`->Equals`/`->Compare`/`->CompareCodePointOrder`** — Priority: **P0**. Status: missing entirely. `BeginsWith`/`EndsWith`/`Find` are extremely common (URL-routing dispatch, extension checks, prefix stripping) and their absence forces awkward `->contains` workarounds that don't distinguish position. Location: `Evaluator.swift` member switch.
- **`String->PadLeading`/`->PadTrailing`/`->Remove`/`->RemoveLeading`/`->RemoveTrailing`/`->Reverse`/`->Merge`** — Priority: **P1**. Status: missing. `PadLeading`/`PadTrailing` are used constantly for zero-padded IDs/codes; `RemoveLeading`/`RemoveTrailing` for trimming a specific character (not whitespace, which `->trim` already covers). Location: `Evaluator.swift`.
- **`String->Get`** (single-character-by-position) — Priority: **P1**. Status: missing (only whole-string `->substring` exists). Location: `Evaluator.swift`.
- **String validation family**: `->IsAlpha`, `->IsAlphaNumeric`, `->IsDigit`, `->IsHexDigit`, `->IsLower`, `->IsNumeric`, `->IsPunctuation`, `->IsSpace`, `->IsUpper` (Ch 25, Table 7/8) — Priority: **P1**. Status: missing entirely. Used for input validation (form field sanity checks) independent of the already-implemented `Valid_Email`/`Valid_CreditCard`. Location: `Evaluator.swift`, likely backed by a new `LassoStringValidation` helper mirroring the existing `LassoValidation.swift` pattern.
- **`String->Foldcase`/`->Titlecase`/`->toLower`/`->toUpper`/`->toTitle`** (per-character case ops, Ch 25 Table 5) — Priority: **P2**. Status: missing (only whole-string `uppercase`/`lowercase` exist, no `titlecase`).
- **Character-information family**: `->CharName`, `->CharType`, `->CharDigitValue`, `->Digit`, `->GetNumericValue`, `->IsAlnum`, `->IsBase`, `->IsCntrl`, `->IsPrint`, `->IsTitle`, `->IsWhitespace`, `->IsUAlphabetic`/`->IsULowercase`/`->IsUUppercase`/`->IsUWhiteSpace` (Ch 25 Table 11) — Priority: **P3**. Status: missing entirely. Unicode-property introspection; genuinely obscure outside text-processing tools.
- **Free-tag substitution family**: `String_Concatenate`, `String_Insert`, `String_Remove`, `String_RemoveLeading`, `String_RemoveTrailing`, `String_Replace`, `String_UpperCase`, `String_LowerCase`, `String_Extract`, `String_FindBlocks`, `String_FindPosition`, `String_Is*`, `String_Length`, `String_EndsWith`, `String_GetUnicodeVersion`, `String_CharFromName` — Priority: **P1** for `String_Length`/`String_Replace`/`String_Extract`/`String_FindPosition` (near-universal legacy-Lasso-8 spelling of the member-tag equivalents — the codebase's own `Documentation/legacy-define-tag-type-plan.md` note confirms legacy code favors this call style), **P2** for the rest. Status: entirely missing (verified — zero `register("string_...")` calls exist). This whole free-tag dialect is absent even though the codebase already has a precedent for exposing one operation two ways (`LassoEncoding`/`LassoDateFormatting` shared-implementation-multiple-callers pattern). Location: new `Sources/LassoParser/StringOperations.swift` housing the shared implementations, called from both the `Evaluator.swift` member cases and new `Runtime.swift` free-tag registrations — directly matching the `Encrypt_HMAC`/`Currency`/`Percent` precedent in `outstanding-compatibility-project-plans.md`.
- **`String_ReplaceRegExp`/`String_FindRegExp`/`String_ExtractRegExp`** — see Regular Expressions section below; these depend on the regex engine that doesn't exist yet at all.

## 3. Regular Expressions (Chapter 26) — entirely missing

**Priority: P0.** Status: missing entirely — no `RegExp` type, no `Match_RegExp`/`Match_NotRegExp`/`Match_Range`/`Match_Comparator` matchers, no `String_ReplaceRegExp`/`String_FindRegExp` tags, no regex-capable string members at all. This is a striking gap: the Language Guide's *own* HTTP/HTML chapter (Ch 49, page 614-615) demonstrates using `String_ReplaceRegExp` to post-process an included remote page, and the Control Tags chapter's own `Define_AtEnd` HTML-cleanup example (page 565) relies on `String_ReplaceRegExp` for whitespace normalization — regex is treated as ordinary, load-bearing infrastructure throughout the guide's own worked examples, not a specialty feature. Given real Lasso 8 sites lean on regex for URL parsing, input sanitization, and template post-processing, this is very likely to be hit by any deeper corpus crawl and was probably masked so far only because whatever calls it happen to sit behind other unresolved gaps (a `String_ReplaceRegExp` call inside a page that already fails on an earlier construct never gets reached).

Rationale/scope: `NSRegularExpression` (Foundation) is available on both Apple platforms and Linux via `swift-corelibs-foundation`, so this is implementable without new external dependencies — consistent with the project's stated Linux-CI investment. Needs: a `RegExp` value/native type (`\g` group indexing via `\\0`-`\\9` replacement placeholders per Ch 26 Table 5), the five documented matchers, and `String->Find`/`->ReplaceRegExp`/`->FindRegExp` members plus the free-tag equivalents.

Suggested location: new `Sources/LassoParser/RegularExpressions.swift` (a `LassoRegexEngine` wrapping `NSRegularExpression`, translating Lasso's `\d`/`\w`/`\p{...}` escape conventions — confirmed compatible with ICU regex syntax that `NSRegularExpression` already uses, so translation should be minimal), registered as a native type in `NativeTypes.swift` (mirroring the `date`/`bytes` pattern) plus member/free-tag hookup in `Evaluator.swift`/`Runtime.swift`.

## 4. Array / Map / Compound Data Types (Chapter 30) — the largest confirmed gap

This is the single biggest finding of the review. Chapter 30 documents eleven distinct compound-data-type concepts; **only Array and Map exist, and even those implement a small fraction of their own documented member surface.**

**Array** — current surface: `size`, `first`, `insert`, `get` (4 members). Missing (Ch 30 Table 2, ~24 documented members): `->Contains`, `->Difference`, `->Find`, `->FindPosition`, `->ForEach`, `->InsertFirst`, `->InsertFrom`, `->InsertLast`, `->Intersection`, `->Iterator`, `->Join`, `->Last`, `->Merge`, `->Remove`, `->RemoveAll`, `->RemoveFirst`, `->RemoveLast`, `->Reverse`, `->ReverseIterator`, `->Reserve`, `->Second`, `->Sort`, `->SortWith`, `->Union`. Priority: **P0** for `->Join`/`->Remove`/`->RemoveAll`/`->Sort`/`->Last`/`->Find` (these are used in virtually every real Lasso template that builds an HTML list, CSV export, or dropdown from query results — this codebase's own `Array->First` fix, noted in the compatibility matrix, was needed for exactly one real corpus call site; `->Join`/`->Sort` will be needed at least as often once a corpus exercises more pages). **P1** for the rest.

**Map** — current surface: bare key lookup only (`values[normalized] ?? .null`) plus `->insert`. Missing (Ch 30 Table 7): `->Find` (the *documented* accessor — the codebase's bare-key-lookup fallback is a non-standard shortcut that also means **any** method name not literally `"insert"` gets swallowed as a key lookup, silently masking typos), `->Get`, `->Keys`, `->Remove`, `->Size`, `->Values`. Priority: **P0** — `->Keys`/`->Values` are the only documented way to enumerate a map's contents outside `Iterate`, and `->Size` collision with the bare-key-fallback means `$myMap->Size` currently returns `.null` (a key named "size") instead of a count if the map happens to have no `"size"` key — a real, silent correctness bug distinct from the missing-feature gaps elsewhere in this list.

**Pair** — `.pair` value exists with `first`/`second` members (matches Ch 30 Table 9) — essentially complete. Not a gap.

**Lists, Priority Queues, Queues, Series, Sets (as a real type), Stacks, Tree Maps, Comparators, Matchers, Iterators** — Priority: **P1** (List/Set/Stack/Queue are common enough — cart/undo-stack/FIFO-processing patterns — to matter once a broader corpus is crawled; Priority Queue/Tree Map/custom Comparators are **P2**). Status: **missing entirely**, confirmed by exhaustive grep (`priorityqueue`, `queue`, `stack`, `treemap`, `series`, `iterator`, `\Compare_LessThan` etc. — zero matches in `Sources/LassoParser`). The codebase's own `Runtime.swift` (line 355-363) has a telling comment: its `set(...)` free function is a **placeholder that returns a plain `.array`**, with an explicit code comment acknowledging "a real gap: the resulting color list can contain duplicates this way" — the implementers already knew Set semantics (uniqueness) weren't real, but shipped the placeholder anyway to unblock one corpus page rather than implement the actual type. This is exactly the kind of thing a top-down pass catches that a corpus pass doesn't: the placeholder makes the corpus page "work" while leaving the entire Set/Stack/Queue/PriorityQueue/TreeMap/Iterator/Comparator/Matcher subsystem (roughly 100 pages of documented tag surface — Ch 30 pages 404-426) completely absent.

Also missing: the `Iterator`/`ReverseIterator` mechanism (Ch 30 pages 422-424) that `While`-loop-based manual iteration depends on (`Iterator->Forward`/`->AtEnd`/`->Value`/`->Key`/`->RemoveCurrent`/`->InsertAtCurrent`) and the five built-in `Comparator`s (`\Compare_LessThan` etc.) and `Match_*` matchers used with `->SortWith`/`->RemoveAll`/`->Iterator`.

Rationale: arrays/maps are the most-used data structures in any real Lasso site (the codebase's own compatibility matrix flags array/map as needed "internally" for `Action_Params`, page variables, and query results) — a member surface this thin means any template doing real list/report manipulation beyond the four already-implemented operations will hit `unsupportedExpression`.

Suggested location: given the codebase's stated architecture (`LassoValue` cases are Swift value types; `.object(LassoObjectInstance)` is the extension point already used for `date`/`bytes`), the cleanest fit is:
- Expand `Evaluator.swift`'s `.array`/`.map` member-dispatch arms directly for the missing Array/Map members (same pattern as the existing four).
- New `Sources/LassoParser/CompoundTypes.swift` implementing List/Set/Stack/Queue/PriorityQueue/TreeMap as native object types backed by `LassoObjectInstance` (mirroring `BytesType.swift`'s `LassoBytesValue.makeObject` pattern), registered in `NativeTypes.swift`.
- New `Sources/LassoParser/Iterators.swift` for the `Iterator`/`ReverseIterator` mechanism and the `Comparator`/`Matcher` constants, since these cross-cut every compound type above and are architecturally distinct from any single type's own member table.

## 5. Variables, References, and the Null/Type System

**`Var_Reset`/`Global_Reset`/`Local_Reset`, `[Reference]`/`@` symbol, `Null->DetachReference`, `Null->RefCount`** (Ch 15, Table 6) — Priority: **P0**. Status: missing entirely (only `var_defined`/`local_defined` exist; no `global_defined`, no reset/reference family at all). This matters because Lasso's documented reference semantics (multiple variables/array-elements/map-elements sharing one underlying value via `@`) are exactly what the codebase's own `-Required`/`-Optional` parameter binding and `Iterate` loop-variable binding *implicitly rely on* per the Guide's own worked example (Ch 15, page 232 — `Ex_Square` custom tag mutating an array element in place through an implicit reference). If this interpreter's parameter-binding doesn't actually implement reference semantics (worth verifying directly, since it's plausible the current implementation only *looks* like it works by coincidence of Swift's copy-on-write `Array`), any custom tag written against the documented reference contract may silently not mutate the caller's data. Location: `Evaluator.swift` (assignment/declare paths already handle `.variable`; needs a genuine reference-value wrapper, not just copy semantics) plus `Runtime.swift` for the free-tag registrations.

**`Globals()`/`Locals()`** (return-all-as-map, Ch 15 Tables 1/3/4) — Priority: **P1**. Status: missing (only per-name `var_defined`/`local_defined` exist, no bulk enumeration).

**`Null->Type`/`->IsA`/`->Properties`/`->FreezeType`/`->FreezeValue`/`->Serialize`/`->Deserialize`/`->XMLSchemaType`** (Ch 43, Table 6) — Priority: **P0** for `->Type` and `->Serialize`/`->Deserialize`, **P2** for the rest. Status: missing entirely. `->Type` (run-time type introspection — `[123->Type]` → `Integer`) is used pervasively in real code for type-dispatch logic (the Guide's own `Select`/`Case` chapter worked example, Ch 16 page 238, dispatches directly on `->Type`) — and the codebase's `Select`/`Case` implementation (per `outstanding-compatibility-project-plans.md`) was *just* built, so any real page using this exact documented pattern will fail today. `->Serialize`/`->Deserialize` matter because the codebase already has a `jsonObject` conversion (`Runtime.swift` extension) that could back a `Null->Serialize`/`->Deserialize` implementation almost for free — this is a case where the missing feature is unusually cheap to add given existing infrastructure. Location: `Evaluator.swift` member dispatch (needs a genuine catch-all "any value" case, not scoped to one `LassoValue` case) plus `Runtime.swift` for the bare `[Null]` literal tag.

**Page Content Tags** (`Content_Body`, `Content_Header`, `Content_Encoding`, `Tags`, `Variables`) — Priority: **P2**. Status: missing (though `Response_Path`-family request tags are partly covered by `web_request` — see below). `Content_Body`/`Content_Header` (low-level response mutation) are documented as needed for exactly the kind of pre/post-processing the codebase's own `Define_AtBegin`/`Define_AtEnd` deferred-flag notes reference.

## 6. Math Operations (Chapter 28)

**Priority: P1.** Status: the arithmetic *symbols* (`+`, `-`, `*`, `/`, `%`, comparisons) work via the expression parser, and `Currency`/`Percent` are implemented, but the entire **Math_\* substitution-tag family is missing**: `Math_Abs`, `Math_Add`, `Math_Ceil`, `Math_ConvertEuro`, `Math_Div`, `Math_Floor`, `Math_Max`, `Math_Min`, `Math_Mod`, `Math_Mult`, `Math_Random`, `Math_RInt`, `Math_Roman`, `Math_Round`, `Math_Sub`, plus the trig/advanced family (`Math_ACos`, `Math_ASin`, `Math_ATan`, `Math_ATan2`, `Math_Cos`, `Math_Exp`, `Math_Ln`, `Math_Log10`, `Math_Pow`, `Math_Sin`, `Math_Sqrt`, `Math_Tan`) and `Scientific`/`Locale_Format` (Ch 28 Table 13 — deferred already, but zero-evidence deferral was made before this pass confirmed the whole family around it is also missing). `Math_Random` in particular is **P0** — random-number generation (order confirmation numbers, CAPTCHA-adjacent tokens, sample/A-B-test selection) is a common real-world need with no substitute currently available in the interpreter at all. `Decimal->SetFormat`/`Integer->SetFormat` (currency/precision formatting, Ch 28 Tables 6-9) and the bit-operation family (`Integer->BitAnd/BitOr/BitXor/BitNot/BitShiftLeft/BitShiftRight/BitClear/BitFlip/BitSet/BitTest`) are **P2** — real but lower-frequency (permission flags, low-level binary math).

Suggested location: `Runtime.swift` for the free-tag family (mirrors the existing `currency`/`percent` registrations exactly), new `Sources/LassoParser/MathOperations.swift` for shared trig/rounding helpers backed by Swift's `Foundation`/`Darwin`/`Glibc` math functions, plus `Evaluator.swift` for `Integer`/`Decimal->SetFormat` and bit-op members.

## 9. Encryption, Serialization, Compression (Chapter 42)

Only `Encrypt_HMAC` is implemented (per `outstanding-compatibility-project-plans.md`). Missing entirely (confirmed by grep — no `cipher`, `compress`, or `serialize`/`deserialize` registrations anywhere):

- **`Encrypt_Blowfish`/`Encrypt_Decrypt`, Cipher tags** (Ch 42 Table 1-4) — Priority: **P1**. Status: missing. Password/session-token encryption beyond HMAC signing is a common real-site need (the Language Guide's own known-defect note — `Valid_CreditCard` mislabeled "ROT-13" — is actually in this exact chapter's neighborhood, per the task's own known-quirk callout, so this chapter's text should be read carefully rather than trusted verbatim when eventually implemented).

- **Compression tags** (`Compress`/`Decompress`, Ch 42) — Priority: **P2**. Status: missing.

Suggested location: `Sources/LassoParser/Hashing.swift` already exists for `Encrypt_HMAC` via swift-crypto — extending it (or a sibling `Encryption.swift`) for Blowfish/cipher support is the natural fit given swift-crypto is already a direct dependency. Serialize/Deserialize belongs in `Runtime.swift`/`Evaluator.swift` reusing `LassoValue.jsonObject`.

## 10. Web / HTTP / Session Infrastructure Beyond What's Tracked

The compatibility matrix tracks `web_request`/`web_response`/session/inline in detail. Not tracked, and confirmed missing:

- **Authentication tags** (`Auth`, `Auth_Admin`, `Auth_Group`, `Auth_User`, `Auth_Custom`, `Auth_Prompt` — Ch 43 Table 1) — Priority: **P1**. Status: missing entirely. HTTP Basic/Digest auth gating is a common pattern for admin sections of real sites, and is architecturally simple (401 + `WWW-Authenticate` header, which `response_status`/header-setting infrastructure already supports).
- **Cache tags — real implementation** — Priority: **P1**. Status: currently a **no-op stub** (`register("cache") { _, _ in .void }` in `Runtime.swift`, and the compatibility matrix explicitly documents `[Cache]` as "implemented as a no-op native" because "this interpreter has no output-caching layer"). This is a real, acknowledged gap, not a silent one — but it's worth flagging as higher-priority than its current "deferred, matches corpus usage" framing suggests, since the Guide documents `Cache`/`Cache_Fetch`/`Cache_Store`/`Cache_Empty`/`Cache_Delete`/`Cache_Exists` as a first-class performance mechanism (Ch 49) that real sites use specifically to reduce database load — exactly the kind of thing that matters more as traffic grows, independent of whether the crawled corpus happens to exercise it today.
- **`Log_*` tags beyond `log_critical`** (Ch 41 — `Log_Detail`, file-log routing/`Log_AddFile`, log-level/destination controls) — Priority: **P1**. Status: missing (only `log_critical` exists; the Control Tags chapter's own `Define_AtBegin` worked example, page 565, uses `Log_Detail` for page-timing instrumentation — a pattern this codebase's own docs reference elsewhere for observability work).
- **`Include_URL`, `HTTP_GetFile`, `FTP_PutFile`/`FTP_GetFile`/`FTP_GetListing`** (Ch 49) — Priority: **P2**. Status: missing entirely. Remote-page fetching/proxying and FTP file transfer are real but less common in a modern deployment (most real integrations would go through the datasource layer instead).
- **Client/Server info tags** (`Client_Address`, `Client_Browser`, `Client_IP`, `Client_Type`, `Client_URL`, `Server_Port`) — Priority: **P1**. Status: partially covered — `web_request->remoteAddr`/`serverName`/`serverPort` exist as native-type members, but the Lasso-8-era free-tag spellings (`Client_IP`, `Client_Browser`, `Server_Port`, etc.) do not, and `web_request` has no `->browser`/`->type` equivalent at all (User-Agent parsing). Given `server_name` was already a confirmed real bug fixed in this codebase (per project memory), the sibling `Client_*`/`Server_*` free-tag family is worth auditing for the same class of issue.
- **`Admin_*` tags** (Lasso Security administration, Ch 43 Table 2) — Priority: **P2**. Status: missing; admin-console-only usage.
- **Scheduling (`Event_Schedule`)** — Priority: **P2**. Status: missing; background-job scheduling, real but infrastructure-dependent (would need the server's own scheduler, not just a parser/runtime addition).
- **`Lasso_TagExists`/`Lasso_Version`/`Lasso_TagModuleName`, `Lasso_DatasourceIs*`, `[Compiler_RemoveCachedDoc]`, `[NoProcess]`** — `lasso_tagexists`/`tag_exists`/`noprocess` are already implemented per `library-and-custom-tags.md`/the compatibility matrix; `Lasso_Version`/datasource-introspection tags are **P3**, low real-world need.

## 11. Namespaces, DNS, Threads, Unique ID (small but genuine gaps)

- **`Namespace_Using`** (Ch 40) — Priority: **P2**. Status: missing. Referenced by the Control Tags chapter's own `Define_AtBegin` worked example (page 564) as the correct way to scope custom-tag registration from a pre-process include — directly relevant to this codebase's existing `library()`/custom-tag-registration machinery.
- **`DNS_Lookup`** (Ch 54) — Priority: **P2**. Status: missing entirely.
- **`Unique_ID`** (Ch 46) — Priority: **P1**. Status: missing entirely. Session/token/order-ID generation is a common real need; cheap to implement (UUID generation).
- **`Thread_Atomic`/Thread tools** (Ch 44) — Priority: **P2**. Status: missing. The Variables chapter's own documented pattern for safely mutating a global array from concurrent requests (Ch 15, page 228) depends on `Thread_Atomic`; this codebase's `LassoTagRegistry` already uses `NSLock` internally, so an equivalent user-facing primitive is architecturally consistent with the existing concurrency model, not a new paradigm.

## 12. Email/POP, XML, and Lower-Priority Chapters (P2/P3)

- **`Email_Send` — currently a no-op stub** (`register("email_send") { _, _ in .void }`) — Priority: **P1**. Status: acknowledged stub, not silently missing (per the code comment cited in the runtime), but worth flagging as higher-value than a P2/P3 item because order-confirmation and password-reset emails are core e-commerce flows this project's own corpus is known to include (`lost_password.page.lasso` already surfaced in project memory). Full POP support (`Email_POP`/`Email_Parse`, Ch 48) is genuinely **P3** — receiving mail is a much rarer server-side need.
- **XML type/`XML_Extract`/XPath** (Ch 34) — Priority: **P2**. Status: missing entirely. Relevant if any real datasource or third-party integration returns XML (the FileMaker XML CWP work already in this codebase parses XML at the transport layer via `Perfect-XML`, but exposes none of it as a first-class Lasso `XML` value/XPath surface to user code).
- **Images (Ch 32), PDF (Ch 35), JavaBeans (Ch 36), iCalendar (Ch 37), Process/Shell (Ch 38), LDAP (Ch 39), XML-RPC (Ch 50), SOAP (Ch 51), Wireless/WAP (Ch 52), AJAX/LJAX (Ch 53)** — Priority: **P3** across the board. All missing entirely. These are legitimately low-value for this project: binary-format generation (PDF/Images), JVM interop (JavaBeans), OS process spawning (Process/Shell — also a real security-surface concern to add deliberately, not by default), directory-service auth (LDAP), and dead/legacy web-service protocols (XML-RPC, SOAP, WAP) are all poor fits for a from-scratch Swift server whose value proposition is compatibility with real e-commerce/CMS page logic, not full parity with Lasso's 2007-era platform-integration surface. Recommend explicit "Unsupported, by design" classification in the compatibility matrix (matching the existing treatment of LCAPI/LJAPI) rather than leaving these as an implicit, undifferentiated backlog.

## Top 10 — Do First (executive summary, ranked by expected real-site impact)

1. **`Loop_Abort`/`Loop_Continue`** (Control Flow) — missing break/continue is a foundational control-flow gap on an otherwise-complete loop implementation; cheap to add given the existing `returnSignal` precedent. `Renderer.swift` + `Runtime.swift`.
2. **Array/Map member-tag expansion** (`->Join`, `->Sort`, `->Remove`, `->RemoveAll`, `->Find`, `->Keys`, `->Values`, `->Size` for Map) — the single largest confirmed gap in the whole review; these are used in essentially every real template that renders a list or report. Also fixes the real `Map->Size` name-collision bug (falls through to key lookup today). `Evaluator.swift`.
3. **Regular expressions** (`RegExp` type, `Match_RegExp`, `String->ReplaceRegExp`/`->FindRegExp`) — entirely absent despite being load-bearing infrastructure in the Guide's own worked examples (HTTP-content post-processing, HTML cleanup); implementable on `NSRegularExpression` with no new dependency. New `RegularExpressions.swift`.
4. **List/Set/Stack/Queue/PriorityQueue/TreeMap + Iterator/Comparator/Matcher** — an entire documented subsystem (Ch 30) with zero implementation; the codebase's own `set(...)` placeholder comment already admits the gap ("a real gap: the resulting color list can contain duplicates"). New `CompoundTypes.swift`/`Iterators.swift`.
5. **String member/free-tag expansion** (`BeginsWith`/`EndsWith`/`Find`/`PadLeading`/`PadTrailing`/`Remove`/`Reverse`, plus the `String_*` free-tag dialect) — roughly 30 of ~50 documented members missing; blocks common prefix/padding/ID-formatting patterns. `Evaluator.swift` + new `StringOperations.swift`.
6. **`Null->Type`/`->Serialize`/`->Deserialize`** — type introspection is used in the Guide's own `Select`/`Case` dispatch pattern (a feature this codebase just built), and Serialize/Deserialize is nearly free given the existing `jsonObject` conversion. `Evaluator.swift` + `Runtime.swift`.
7. **`Var_Reset`/`Global_Reset`/`Local_Reset`/references (`@`, `[Reference]`)** — the documented reference-semantics contract that custom-tag array/parameter mutation patterns rely on; worth verifying the current implementation doesn't just accidentally work via Swift copy-on-write rather than real reference semantics. `Evaluator.swift`.
8. **File_\* tag family + `[File]`/`[Directory]` type** — general filesystem read/write is entirely missing except upload-specific tags; natural extension of existing root-confined include-loader infrastructure. New `FileOperations.swift`.
9. **Math_\* substitution-tag family** (especially `Math_Random`, `Math_Round`, `Math_Sqrt`) — arithmetic symbols work, but the entire substitution-tag dialect and all trig/rounding/random functions are absent; `Math_Random` has no substitute at all today. `Runtime.swift` + new `MathOperations.swift`.
10. **Real `[Cache]` implementation + `Log_Detail`/log-file tags** — both currently no-op/partial stubs (acknowledged in code comments), not silent gaps, but worth prioritizing ahead of most P2 items because caching and logging compound in value as real traffic/operational needs grow, independent of what today's crawled corpus happens to exercise.
