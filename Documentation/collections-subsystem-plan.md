## 1. Full Inventory (Lasso 8.5 Ch. 30, verified page-by-page)

### 1.1 Array (existing, context only ‚Äî not re-scoped here, see the two prior docs)
Complete member surface already documented there; not re-audited.

### 1.2 Map (Table 6/7, pp. 400‚Äì403) ‚Äî existing type, one real semantic gap found
- `[Map]` constructor, `->Find` (key lookup), `->Get` ‚Äî **"Returns a **pair** from the map by integer **position**"** (Table 7, p.401) ‚Äî this is NOT the current codebase's bare-key-lookup fallback; it's position-based, pair-returning, and completely undocumented-yet-different from anything implemented today.
- `->Keys`, `->Insert`, `->Remove`, `->Size`, `->Values` ‚Äî all already implemented (verified below).
- p.400 explicit Note: **"the order of elements in a map is not defined... should never be relied upon"** AND **"Map elements cannot be set using [Map->Get]"** (p.402 Note) ‚Äî Lasso 8.5 explicitly forbids a `Map->Get=` setter.
- **Lasso 9 divergence (real, citable)**: lassoguide.com's `collections.html` documents `map->get=(value, key)` as a real setter. This is a genuine version divergence, not a documentation error ‚Äî 8.5 explicitly forbids it, 9 explicitly adds it. Recommendation: implement Lasso 9's `->Get=`/`->Get` ‚Äî the corpus this project targets skews Lasso 9-era per prior batches ‚Äî but do NOT implement position-based `Map->Get(n)` reads without also resolving the ordering question below.

### 1.3 Pair (Table 8/9, pp. 403‚Äì404) ‚Äî essentially complete, ONE real gap
`->First`/`->Second` are documented as assignable: **"Can be used as the left parameter of an assignment operator to change the first/second element of the pair"** (Table 9), with a worked example: `$Test_Pair->First = 'Last_Name';`. **Verified via direct grep: this is genuinely unimplemented.** `Evaluator.assign` (Evaluator.swift:446‚Äì491) only accepts `.variable`/`.identifier`/`.string`/`.member`-on-`.object` as assignment targets; `.pair` isn't `.object`, so `$Test_Pair->First = X` throws `invalidAssignment` today. This is a real, freshly-confirmed gap the prior two docs only flagged as "P3, rare" ‚Äî it's directly load-bearing for this batch of work because Stack/Queue's `->First=` (below) needs the identical mechanism.

### 1.4 PriorityQueue (Table 10/11, pp. 404‚Äì407) ‚Äî missing entirely
- `[PriorityQueue]` (optional comparator param, defaults to alphanumeric/numeric ascending via `\Compare_LessThan`), `->First` (peek, no mutation), `->Get` (pop+return), `->Insert`/`->InsertLast` (alias), `->Remove`/`->RemoveFirst` (alias, pop+discard), `->Size`.
- **Real semantic gotcha, verified via the Guide's own Note (p.405) and worked example (p.405‚Äì406)**: priority queues pull their next value off the **end** of the internally sorted list. With the *default* `\Compare_LessThan` comparator (ascending sort), that means the **greatest** value comes out first (`->Insert('One')`, `->Insert('Two')` ‚Üí `->First` returns `'Two'`, confirmed p.406 worked example). Using `\Compare_GreaterThan` reverses it to least-first. This is exactly the class of doc-quirk this project's prior batches have repeatedly caught late ‚Äî flagging explicitly now.
- `->Insert` internally re-sorts on every insert (not a lazy sort-on-read) per "When an element is inserted into a priority queue it is automatically placed in the proper position" (p.405).

### 1.5 Queue (Table 12/13, pp. 407‚Äì410) ‚Äî missing entirely, FIFO
`[Queue]`, `->First` (peek), `->Get` (pop FIFO), `->Insert`/`->InsertLast` (alias, push to end), `->Remove`/`->RemoveFirst` (alias, pop+discard), `->Size`. **`->First` is documented as "by reference so the value of the element can be changed"** (p.409), worked example `$myQueue->First = 'Three';` ‚Äî same assignable-getter mechanism as Pair, above.

### 1.6 Stack (Table 17/18, pp. 413‚Äì415) ‚Äî missing entirely, LIFO
`[Stack]`, `->First` (peek most-recent, **assignable by reference**, identical mechanism/worked-example shape to Queue), `->Get` (pop LIFO), `->Insert`/`->InsertFirst` (alias, push), `->Remove`/`->RemoveFirst` (alias, pop+discard), `->Size`.

### 1.7 Series (Table 14, p.410) ‚Äî NOT in the original ask, flagging as a free bonus
`[Series(start, end)]` produces a plain array of the integer/decimal range. The Guide states explicitly: **"The series data type supports the same member tags as the array data type"** ‚Äî i.e., it's not a distinct type at all once constructed, just a constructor sugar over `.array`. Given Array's member surface is already ~complete in this codebase, `Series` is essentially a single new free-function registration (`series(start, end)` building `.array([...])` by incrementing) with near-zero marginal cost. Recommended as a cheap Stage-1 addition even though outside the original nine-type list, since silently dropping a two-line-cost documented feature found mid-pass would violate this project's own disclosure norm.

### 1.8 Set (Table 15/16, pp. 410‚Äì412) ‚Äî missing entirely, the anchor gap
`[Set]` (optional comparator, default alphanumeric/numeric sort), `->Contains`, `->Difference`/`->Intersection`/`->Union` (each takes another `Set`, returns a new `Set`), `->Find` (returns matching subset), `->ForEach` (**"requires a single parameter which is a reference to a tag or compound data type... modifies the set in place"** ‚Äî needs a passable-tag-reference primitive, see ¬ß3.3), `->Get`/`->Insert`/`->Iterator`/`->Join`/`->Remove`/`->RemoveAll`/`->ReverseIterator`/`->Size`. Elements are **always sorted** and **duplicate-checked on insert** (p.410 ‚Äî "A set is a collection of unique values... always sorted... When a new element is inserted the set is first checked to see if the value is already contained"), verified live in the worked example: inserting `'Three'` three times yields `Set: (One, Three)`.

**This is the exact gap `Runtime.swift:629‚Äì641`'s `set(...)` free function admits** (own comment: *"a real gap: the resulting color list can contain duplicates this way"*), currently shipping as a bare `.array(arguments.map { $0.value })` alias with no dedup and no sorting. Confirmed unchanged as of this scoping pass.

### 1.9 TreeMap (Table 19/20, pp. 416‚Äì419) ‚Äî missing entirely
Same member surface as Map (`->Find`/`->Get`(position‚Üípair)/`->Insert`/`->Keys`/`->Remove`/`->RemoveAll`/`->Size`/`->Values`) plus `->Iterator` (with optional key-matcher). Two real, documented distinctions from plain Map: **(1)** "keys in a tree map can be **any Lasso data type**. In a simple map all keys are converted to string values" (p.416) ‚Äî real Map here coerces every key to `String` via `outputString`; TreeMap must NOT. **(2)** "the keys in a tree map can be **sorted** using a comparator which is provided when the tree map is created" (p.416) ‚Äî genuinely ordered-by-key storage, unlike Map's explicitly-undefined order.

### 1.10 Comparators (Table 21, p.419) ‚Äî the sort/compare vocabulary
Built-ins: `\Compare_LessThan` (default for PriorityQueue), `\Compare_GreaterThan`, `\Compare_Contains`, `\Compare_NotContains`, `\Compare_EqualTo`, `\Compare_NotEqualTo`, `\Compare_StrictEqualTo` (type-strict), `\Compare_StrictNotEqualTo`. **Comparators return an integer, not a boolean ‚Äî 0 signals a valid/matching comparison, any other value signals no match** (explicit Note, p.419) ‚Äî a real, easy-to-invert-by-accident semantic. Custom comparators (p.420) are built with an **ordinary `Define_Tag`** taking `-Required='Left'`/`-Required='Right'` params returning an integer ‚Äî NOT a Capture/block literal. (The worked example itself, `If: #Left->First < #Left->Second`, compares `Left`'s own two halves rather than `Left` vs `Right` ‚Äî reads as a likely copy/paste artifact in the Guide's own example; flagging as a probable documentation defect rather than following it literally, matching this project's established practice, e.g. the earlier `Math_Div` finding.)

### 1.11 Matchers (Table 22, pp. 420‚Äì422) ‚Äî distinct from, but touches, the deferred RegExp Table 10 work
Literal string/decimal/integer values act as matchers directly (equality, with type coercion matching `==`). Built-ins: `[Match_RegExp]`/`[Match_NotRegExp]` (take a regex pattern ‚Äî **this is a NEW, separate mechanism from RegExp Table 10's "Interactive Find/Replace" tags** deferred in `RegularExpressions.swift`'s own doc comment; Match_RegExp/NotRegExp are stateless one-shot predicate matchers, not the stateful position-advancing `->Find`/`->MatchString`/`->AppendReplacement` family that was actually deferred ‚Äî **confirmed, via direct read of both sources, these do NOT overlap**, so this is genuinely new scope, not already-covered-and-mislabeled work), `[Match_Range]`/`[Match_NotRange]` (low/high bounds), `[Match_Comparator]` (wraps a Comparator with `-RHS`/`-LHS`). Matchers, like Comparators, return an integer (0 = match) not a boolean (same Note pattern, p.420). Custom matchers (p.422) are built via `Define_Type` + `Define_Tag: 'onCompare', -Required='Value'` ‚Äî an auto-invoked callback hook name, not currently supported by this codebase's dispatch at all (confirmed: zero `oncompare` hits anywhere in `Sources/LassoParser`).

The `>>` (contains) operator and `->Contains`/`->RemoveAll`/`->Iterator` across every compound type accept a Matcher as their parameter (p.420‚Äì421 worked examples: `(Array:1..7) >> 7` ‚Üí `True`; `>> (Match_Range: 1, 4)` ‚Üí `True`). **Adjacent finding**: `>>`'s current implementation (`Evaluator.swift:553-566`) is pure `left.outputString.contains(right.outputString)` ‚Äî it doesn't branch on `.array` at all today, so `(Array:1,2,3) >> 7`-style array-membership already silently degrades to string-concatenation-then-substring-search (works by coincidence for short/single-token elements, breaks in general). This predates this scoping pass but is directly touched by Matcher work and should be fixed alongside it.

### 1.12 Iterators (Tables 23/24, pp. 422‚Äì426) ‚Äî cross-cutting, genuinely mutable
`[Iterator]`/`[ReverseIterator]` (built-in-preferred, generic `->Size`/`->Get`-based fallback otherwise), returned by `->Iterator`/`->ReverseIterator` on array/list/map/set/treemap. Member tags: `->Forward`/`->Backward`/`->Left`/`->Right`/`->Up`/`->Down` (built-ins only ever support forward/backward ‚Äî left/right/up/down always report `False`/no-op, reserved for future multi-dimensional custom types), `->AtEnd`/`->AtBegin`/`->AtFarLeft`/`->AtFarRight`/`->AtTop`/`->AtBottom`, `->Reset`, `->Key` (map/treemap only), `->Value` ("returns a **reference** to the value of the current element" ‚Äî explicitly reference semantics, p.424), `->RemoveCurrent`, `->InsertAtCurrent`. **An iterator inherently carries mutable per-call position state tied to the specific collection instance it was created from** ‚Äî architecturally this is the same shape as the already-deferred RegExp Table 10 stateful tags (`->Find` advancing a match position), and should be treated with the same "needs genuine mutable per-call state, not the value-semantics convenience-tag pattern" caution documented there.

### 1.13 Lasso 9 cross-reference (lassoguide.com `operations/collections.html`, fetched directly)
Confirms List/Set/Stack/Queue/Map/Array/StaticArray exist in Lasso 9's leaner model, with camelCase method names (`insertFirst()`, `removeAll(matching=?)`, etc.) that already match this codebase's lowercase-normalized dispatch convention. **PriorityQueue, TreeMap, and the formal Iterator/Comparator/Matcher vocabulary are NOT present anywhere in Lasso 9's collections page or language-guide TOC** ‚Äî Lasso 9 replaced that explicit iteration/comparison apparatus with query expressions (`with...in...where...select`, already tracked as blocked-on-Captures in the prior gap doc) and an implicit "elements must be able to `onCompare` themselves" contract for map keys/set elements. This is a genuine, confirmed 8.5‚Üí9 simplification, not a doc gap on either side ‚Äî recommend building the fuller 8.5 Ch. 30 surface (this project's primary source throughout) since prior batches' corpus is 8.5/9-transition-era code that plausibly still uses the 8.5 vocabulary.

## 2. Current Implementation Cross-Reference (grep/read-verified, not assumed)

- **`set(...)`** ‚Äî `Runtime.swift:629‚Äì641`. Confirmed: plain `.array` alias, comment explicitly admits no dedup. The anchor gap for this whole batch.
- **`.array`/`.map`/`.pair` in `Evaluator.member`** ‚Äî `Evaluator.swift:852‚Äì1069`. Array's documented surface is essentially complete (size/first/insert/get/last/second/reverse/sort/join/contains/find/findposition/remove/removeall ‚Äî all present and cited to Ch. 30 page numbers in their own doc comments). Map has `insert`/`remove`/`removeall`(undocumented convenience)/`size`/`keys`/`values`/`contains`/`find` plus a **key-first-with-fallback** dispatch design (`.map(values), _` where `values[normalized] != nil` wins before any named method, except `insert`/`remove`/`removeall` which stay unconditional to avoid the `evaluateStatement` self-mutating write-back silently clobbering a whole map variable on a key collision ‚Äî Evaluator.swift:1005-1029 comment). Map's `->keys`/`->values` deliberately iterate in **sorted-by-key order**, not raw Dictionary order, specifically because "the order of elements in a map is not defined" per the Guide itself (Evaluator.swift:1032-1043 comment) ‚Äî this is the exact precedent to reuse for `Map->Get(n)`/`TreeMap` ordering decisions (¬ß3 below). Pair has only `first`/`second` (read-only) ‚Äî no setters, confirmed above.
- **`LassoValue`** (`Runtime.swift:10-88`) ‚Äî `indirect enum`, already `Equatable, Sendable`. `.object(LassoObjectInstance)` is the existing reference-typed extension point. **No `.capture` case exists.**
- **`LassoObjectInstance`** (`TypeSystem.swift:3-34`) ‚Äî `final class`, `@unchecked Sendable, Equatable` (via `===`, reference identity), a `NSLock`-protected `[String: LassoValue]` bag with `value(for:)`/`set(_:for:)`/`snapshotData()`. This is the class every native type (`date`/`bytes`/`regexp`/`web_request`/`web_response`) is built on.
- **`LassoNativeType`/`LassoNativeTypeRegistry`** (`NativeTypes.swift:21-71`) ‚Äî the established registration pattern: a flat `[String: LassoNativeMethod]` table per type name, registered once in `registerDefaultTypes()`. This is the template List/Set/Stack/Queue/PriorityQueue/TreeMap should follow.
- **Critical existing precedent, `NativeTypes.swift:399-489` (`makeDateType`) and `:528-547` (`makeRegExpType` doc comment)**: this codebase's `.object`-wrapped native types **deliberately implement copy-on-assignment value semantics**, not reference/aliasing semantics, DESPITE being backed by a Swift class. `Date->Add`/`->Subtract` explicitly never mutate `receiver` ‚Äî they build a brand-new `LassoObjectInstance` and return it; the "looks like mutation" effect for a bare-statement call comes entirely from `Evaluator.evaluateStatement`'s `selfMutatingMethods` (`Evaluator.swift:1175-1198`) write-back mechanism reassigning the CALLING VARIABLE, never touching any other variable that happens to alias the same old instance. This was a real, found-and-fixed bug (an earlier `Date->Add` draft mutated `receiver` directly and silently corrupted `var(d2=$d1)` after `$d1->add(...)`). **`selfMutatingMethods` is purely name-based (syntactic), not type-scoped** ‚Äî so extending it for new type-specific mutating method names (`insertfirst`, `insertlast`, `removefirst`, `removelast`, `sortwith`, etc.) is a trivial, safe addition with no cross-type interference risk.
- **`Evaluator.assign`** (`Evaluator.swift:446-491`) ‚Äî accepts `.variable`/`.identifier`/`.string`/`.member`-on-`.object` (the last one just calls `object.set(value, for: name)`, a generic property write with no type-specific behavior) as assignment targets. **Does not special-case any member name** ‚Äî confirms Pair/Stack/Queue's `->First=`/`->Second=` genuinely need new logic here, not just "falls out for free" from being `.object`-backed (a naive `.object` case write would store a literal `"first"` data key rather than actually replacing the head of whatever internal element-list representation the getter reads from).
- **`lassoEquals`/`lassoSortKey`/`lassoLessThan`** (`Evaluator.swift:1200-1245`) ‚Äî real, already-battle-tested infrastructure this plan builds directly on. `lassoEquals` routes through the same case-insensitive `binary(_:"==",_:)` used everywhere else (fixing a real production incident cited in its own comment ‚Äî `'Yes'` vs `'yes'`). `lassoSortKey`/`lassoLessThan` implement a genuine strict-weak-ordering natural sort (numeric-first, then lexicographic) already used by `Array->Sort`, with documented edge-case handling (`NaN`/`Infinity` string tokens). **This is Set's default comparator, PriorityQueue's default comparator, and TreeMap's default key-order ‚Äî all for free, no new equality/ordering infrastructure needed for the natural-ordering case.**
- **`Equatable`/`Hashable` audit** (exhaustive grep) ‚Äî `LassoValue` and `LassoObjectInstance` are both `Equatable` (the latter via reference identity `===`, deliberately, per its own design ‚Äî two different `date` objects with identical field values are NOT `==` under this operator, only `lassoEquals`'s semantic-comparison path treats them as equal via `.outputString`/`binary("==")`). **Nothing is `Hashable` anywhere in `LassoParser`.** Set dedup should therefore use **linear scan + `lassoEquals`** (matching `Array->Contains`/`->RemoveAll`'s existing O(n) pattern), NOT a Swift `Set<LassoValue>`/dictionary-keyed approach ‚Äî consistent with existing code, and correctly avoids having to invent a `Hashable` conformance that would need to agree with `lassoEquals`'s case-insensitive/type-coercing equality (a real and easy-to-get-wrong hashing/equality-consistency requirement that the existing linear-scan design sidesteps entirely at a real but acceptable O(n¬≤) worst-case cost for typical corpus set sizes).
- **`ExpressionParser.swift`** ‚Äî the only backslash handling anywhere (line 81) is string-literal escape-sequence parsing. **No bareword `\identifier` literal syntax exists.** `\Compare_LessThan` etc. need genuinely new parser support.
- **Captures** ‚Äî confirmed absent (zero `LassoValue.capture`/`case capture` hits), consistent with prior docs.

## 3. Architectural Decisions

### 3.1 List/Set/Stack/Queue/PriorityQueue/TreeMap: `.object`-wrapped native types, value-semantics-on-assignment (extend the `date`/`bytes`/`regexp` pattern, not `.array`/`.map`)

**Decision: implement all six as `LassoNativeType`s backed by `LassoObjectInstance`, storing their element collection under a private-by-convention data key (e.g. `_elements`), following the `date`/`regexp` non-mutating-receiver + `selfMutatingMethods` write-back pattern exactly.**

Rationale: the Guide's Ch. 30 text never claims these types have aliasing/reference assignment semantics ‚Äî every worked example (`Var:'myStack'=(Stack)` then later reassigned) shows only per-variable state, and the one place the Guide DOES use the word "reference" is narrowly scoped to `->First`'s return value being a settable slot (¬ß3.6, not general variable-aliasing), and Iterator's `->Value` (¬ß3.2/¬ß1.12, genuinely different ‚Äî an iterator is inherently tied to one specific traversal). Extending `.array`/`.map` (the value-type `LassoValue` cases) directly was considered and rejected: these six types have distinct construction/insertion contracts (sortedness for Set/PriorityQueue/TreeMap, FIFO/LIFO for Queue/Stack, front/back-only access for List) that don't map cleanly onto raw `[LassoValue]`/`[String:LassoValue]`, and ‚Äî critically ‚Äî this codebase already has an established, tested, doc-comment-explained precedent for exactly this shape of type (`date`, `regexp`) that a from-scratch `.array`/`.map` extension would have to reinvent. Using the existing pattern also means `selfMutatingMethods` (already type-agnostic) needs zero structural changes, just new registered names.

### 3.2 Iterator: genuinely reference-typed, but scoped like the RegExp Table 10 deferral

Iterator is architecturally different from the six collection types above: it MUST hold live mutable position state tied to one specific traversal of one specific collection instance ‚Äî the Guide's own three-part definition (collection, position, current value, p.422) is inherently stateful, and `->Forward`/`->Backward`/`->RemoveCurrent`/`->InsertAtCurrent` only make sense with real mutation of the iterator's own internal cursor. **Recommendation: still back it with `LassoObjectInstance` (reusing the same class), but WITHOUT the value-semantics discipline used elsewhere ‚Äî an Iterator's `->Forward` etc. genuinely mutate `receiver` in place** (calling `object.set(...)` on itself), which is safe specifically because ‚Äî unlike `date`/`regexp` ‚Äî no real Lasso code is expected to alias one Iterator into two variables and expect independent cursors; the Guide's own worked examples never do this. This mirrors the RegExp Table 10 stateful tags' own deferred design note (`RegularExpressions.swift:1-15`) almost exactly, and should ship in the SAME stage as the six collection types (it's a shared cross-cutting mechanism they all need for `->Iterator`, not optional polish) ‚Äî not deferred, unlike Table 10 itself.

### 3.3 Comparator: does NOT require Captures ‚Äî needs a smaller, new `\TagName` bareword-reference primitive instead

Directly answering the task's framing: **neither is strictly true.** The built-in comparators (`\Compare_LessThan` etc., Table 21) need zero Captures ‚Äî they're eight fixed, named constants that can ship as a first stage using exactly the existing `lassoLessThan`/`lassoSortKey`/`lassoEquals` machinery, wrapped as small `.object(LassoObjectInstance(typeName:"comparator", ...))` values carrying an enum tag. This closes Set's default ordering, PriorityQueue's default ordering, and TreeMap's default key ordering with NO new parser work.

Custom comparators (p.420) are a different story, but a smaller one than Captures: the Guide's own worked example builds them with an **ordinary `Define_Tag`**, then presumably passes a *reference to that already-defined, already-named tag* wherever a comparator is expected. Lasso 8.5 predates Lasso 9's Captures (`{...}` block literals) entirely ‚Äî the mechanism it actually uses for "pass a callable thing as a value" is the SAME `\identifier` bareword syntax already needed for `\Compare_LessThan` (Table 21 lists these as bare `\`-prefixed names, not tag calls) ‚Äî i.e., **`\TagName` is very likely Lasso 8.5's general "reference to a named, already-defined tag" literal**, not a comparator-specific construct. Recommend scoping this precisely: (1) new lexer/parser support for a `\identifier` expression producing a small new value shape (e.g. `.object(LassoObjectInstance(typeName:"tagreference", data:["name":.string(...)]))`); (2) a generic "invoke this tag reference with N positional/labeled arguments" evaluation path that looks the name up in the SAME tag registry ordinary custom-tag calls already use. This is real, new, but meaningfully smaller and lower-risk than full anonymous-block Captures ‚Äî no new AST node for block bodies, no closure-capture-of-enclosing-scope semantics to design, just "a value that names an existing tag, resolved and invoked later." **Recommendation: ship built-in comparators in the main collections stage; scope custom comparators (`\TagName` + tag-reference invocation) as a clearly separate, smaller follow-on stage, explicitly NOT gated on Captures.**

### 3.4 Matcher: genuinely new work, NOT a duplicate of the deferred RegExp Table 10 tags

Directly verified via both sources (¬ß1.11): Match_RegExp/Match_NotRegExp/Match_Range/Match_NotRange/Match_Comparator are one-shot, stateless predicate values consumed by `->Contains`/`->RemoveAll`/`->Iterator`/`>>` ‚Äî they do not overlap with RegExp Table 10's stateful "Interactive Find/Replace" position-advancing tags (`->Find`/`->MatchString`/`->AppendReplacement`), which remain correctly, separately deferred exactly as `RegularExpressions.swift`'s own doc comment states. Implement Matchers as their own small `.object`-wrapped value family (mirroring the Comparator design in ¬ß3.3), consumed by extending `>>` and the relevant `->Contains`/`->Find`/`->RemoveAll`/`->Iterator` implementations to check "is the argument a matcher object?" before falling back to plain-value equality ‚Äî this is also the natural place to fix `>>`'s pre-existing, adjacent bug (¬ß1.11) where array-membership silently degrades to string-concatenation-then-substring-search. Custom matchers via `Define_Type`+`onCompare` need a genuinely new "well-known callback name auto-invoked by the runtime during a builtin operation" dispatch mechanism that doesn't exist anywhere in this codebase today (confirmed zero `oncompare` hits) ‚Äî flag this as the single riskiest specific sub-item in the whole plan (see ¬ß6).

### 3.5 Set duplicate detection: linear scan + `lassoEquals`, no new Hashable infrastructure

Per ¬ß2's audit: nothing in this codebase is `Hashable`, and inventing a hash function that's guaranteed consistent with `lassoEquals`'s case-insensitive/type-coercing equality is real, non-trivial, easy-to-get-subtly-wrong new infrastructure (e.g. `.string("Yes")` and `.string("yes")` must hash identically). **Recommendation: Set's `->Insert` does a linear `contains(where: lassoEquals)` check before appending** (same cost model `Array->Contains`/`->RemoveAll` already accept), keeping the element list sorted via `lassoLessThan` (or a supplied Comparator, ¬ß3.3) on each insert. This directly closes the `set(...)` placeholder's own admitted gap using entirely existing, tested equality/ordering infrastructure ‚Äî zero new Equatable/Hashable work required.

### 3.6 Assignment-as-target extension for `->First=`/`->Second=` (Pair, Stack, Queue)

This is real, new work in `Evaluator.assign`, not something that falls out of `.object`-backed types automatically (¬ß2). Recommendation: add a new case to `assign`'s switch, ahead of the generic `.object` case, matching `.member(base, name, _)` where `name` is one of a small type-scoped allowlist (`"first"`/`"second"` on `.pair`; `"first"` on Stack/Queue's registered typeName) ‚Äî each writing through to the ACTUAL underlying storage representation (rebuild the pair/replace-head-of-`_elements`), not a generic `object.set(value, for: "first")` data-key write. This is the same class of "genuinely new architectural pattern, not a routine native-type addition" the task's risk-assessment ask is fishing for (see ¬ß6) ‚Äî it's a real extension of `assign`'s target-shape vocabulary, the first since `.object` support was added.

## 5. Deferred, With Reasoning

- **Custom Matchers via `Define_Type`+`onCompare`** (part of Stage 6) ‚Äî needs a "well-known callback name the runtime auto-invokes during a builtin operation" dispatch mechanism that doesn't exist anywhere in this codebase (confirmed zero hits). This is structurally different from every other item in this plan (which are all either new native-type method tables or new parser literals) ‚Äî it's a new *implicit dispatch protocol*, closer in kind to the still-unimplemented trait/operator-overloading gap the lasso9 doc's ¬ß6 already flags (`onConvert` works, but no builtin operation auto-checks for a matching user-defined callback). Recommend treating this as its own scoping question once Stage 6's `\TagName` mechanism actually exists and its real design shape is clearer.
- **List/Set/TreeMap `->ForEach`/`->SortWith`/`->InsertFrom` in full generality** (i.e., accepting an arbitrary user tag reference, not just a built-in Comparator) ‚Äî gated on Stage 6, same reasoning as custom Comparators.
- **Left/right/up/down multi-dimensional Iterator movement** ‚Äî the Guide itself says built-in types never support this (always `False`, p.424); genuinely reserved for hypothetical future custom types, not real scope for this project.
- **`Array + array`, `List/Set->Union` as a general `+`-operator overload on compound types** ‚Äî not documented for Lasso 8.5's Ch. 30 types specifically (Union is `->Union(otherSet)`, not `+`); no gap to close here beyond what's already tracked in the lasso9 doc's ¬ß2 for Lasso 9-specific `array->+(rhs::trait_forEach)`.
- **StaticArray** ‚Äî genuinely adjacent (documented alongside List/Set/Stack/Queue in Lasso 9's `collections.html`) but already tracked as its own distinct item (P1, needs new literal parser support for `(: ...)`) in `lasso9-lassoguide-gap-analysis-plan.md` ¬ß2 ‚Äî not duplicated into this plan.
- **Query Expressions / `with...in...where...select`** ‚Äî already tracked as blocked on Captures in the prior gap doc; this plan's Comparator/Matcher work does NOT unblock it (query expressions are a Lasso 9-only mechanism per ¬ß1.13, orthogonal to the 8.5 Ch. 30 vocabulary this plan targets).

## 6. Risk Assessment ‚Äî where real design judgment (not routine native-type addition) is needed

1. **Highest risk: custom Matchers via `onCompare` auto-dispatch** (¬ß5) ‚Äî needs a genuinely new implicit-callback-invocation architecture, not a template extension of anything existing.
2. **Real but bounded risk: the `\TagName` bareword literal + tag-reference-as-a-value mechanism** (¬ß3.3, Stage 6) ‚Äî new parser/lexer surface and a new value shape, but scoped much smaller than Captures (no anonymous block bodies, no closure-capture semantics) ‚Äî should not be conflated with the Captures gap in review, they're different-sized problems that happen to look similar at a glance.
3. **Real, contained risk: `Evaluator.assign`'s new `.member`-target case** (¬ß3.6, Stage 4) ‚Äî the first extension of assignment-target shapes since `.object` support landed; needs careful per-type write-through logic (not a generic property write) to avoid silently corrupting the underlying collection representation instead of actually replacing the head element.
4. **Moderate risk, single easy-to-invert-backwards detail**: PriorityQueue's greatest-first-by-default-comparator behavior (¬ß1.4) ‚Äî exactly the shape of bug this project's architect-review process has caught before (`Math_Div`, `Array->Remove`/`->RemoveAll` swap); must be verified against the p.406 worked example specifically, not assumed from the comparator's name.
5. **Low risk, routine**: List/Stack/Queue/Series/built-in-Comparator/Iterator implementation ‚Äî all cleanly fit the already-established `date`/`bytes`/`regexp` `LassoNativeType` template with no new architectural pattern required.
