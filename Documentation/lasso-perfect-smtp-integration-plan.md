# LassoPerfectSMTP ‚Äî Integration Plan

Date: 2026-07-19
Status: **Reviewed once (architecture/Lasso-runtime-accuracy, concurrency, security), corrected, not yet implemented.** This document is the output of a research pass against both `/Users/timtaplin/Perfect-Lasso` and `/Users/timtaplin/Perfect-Resurrection/Perfect-SMTP`, an architecture draft, and a full parallel review pass that found one architecturally load-bearing gap in the original draft ‚Äî see ¬ß4.0, added in response, and ¬ß9 (review record) for the full history.

## 1. Why this exists, and what it is not

Perfect-SMTP (`/Users/timtaplin/Perfect-Resurrection/Perfect-SMTP`) is a complete, from-scratch Swift 6.2/SwiftNIO SMTP client ‚Äî implemented across six phases, each independently milestone-reviewed (architecture, concurrency, protocol, security), fully merged to `main`. It was explicitly built to have **zero knowledge of Lasso**: no Lasso-shaped types, no dash-param API, no target inside its own repo for a Lasso adapter. Its own plan document (¬ß4.1, ¬ß6) reserved that work for a future, separate task living entirely inside the Perfect-Lasso repository ‚Äî the same pattern already established by `LassoPerfectFileMaker` and `LassoPerfectCRUD`, which wrap `Perfect-FileMaker`/`Perfect-CRUD` from the outside, as plain consumers of those libraries' public APIs, with zero Lasso-awareness leaking back into the wrapped library.

This document is that future task. It plans a new `LassoPerfectSMTP` target inside Perfect-Lasso that gives `lasso-perfect-server` real, working `email_send`/`email_smtp`/`email_compose`/`email_mxlookup`/`email_result`/`email_status` support, replacing the current no-op stub.

**Scope boundary, restated because it matters:** nothing in this plan modifies Perfect-SMTP's *public API surface as a Lasso-shaped thing* ‚Äî Perfect-SMTP gains no Lasso types, no dash-param handling, no knowledge Lasso exists. It does, however, turn out that this plan's own dispatch-registration problem (¬ß4.0) requires new, generic, Lasso-agnostic protocol surface inside `LassoParser` itself (not Perfect-SMTP) ‚Äî that is a correction from this plan's first draft, found during review, not a violation of the "zero Lasso knowledge" boundary, which only ever applied to Perfect-SMTP.

## 2. Current state, verified directly against both codebases

**Perfect-Lasso's existing adapter precedent.** `LassoPerfectFileMaker`, `LassoPerfectCRUD`, and `LassoPerfectSession` are all single-file SPM library targets under `Sources/`, each depending on exactly `LassoParser` plus the one resurrected library product they wrap (`Package.swift:32-59`). The FileMaker/CRUD adapters implement `LassoDynamicQueryExecutor` ‚Äî **corrected during review: this is a one-method protocol** (`Providers.swift:993-995`: `func execute(_ request: LassoInlineRequest) async throws -> LassoInlineFrame`), not two as this plan's first draft claimed ‚Äî that plugs into the **hardcoded** `"inline"` case in `RendererEngine.render` (`Renderer.swift:252-263`) via a `(any LassoInlineProvider)?` slot on `LassoContext` (`Runtime.swift:1079`). `LassoInlineProvider` is likewise one method (`executeInline`, `Providers.swift:989-991`). Neither adapter "registers a new tag" ‚Äî they both reuse the one existing `[inline]` slot, populated per-request from `main.swift`. `LassoPerfectSession` follows the identical shape via its own `sessionProvider` slot.

**This "reuse an existing context slot" pattern is the ENTIRE precedent this codebase has for wiring an outside target into the interpreter ‚Äî and it does not cover what this plan needs.** See ¬ß4.0.

**Tag/method dispatch is not one registry ‚Äî it's three separate mechanisms:**
- **(a) Free-function tags**, via `LassoNativeRegistry` (`Runtime.swift:98-116`) ‚Äî `register(name, function:)` where `function` is `@Sendable (arguments: [EvaluatedArgument], context: inout LassoContext) async throws -> LassoValue`. This is where the existing `email_send` stub already lives, registered inside `LassoNativeRegistry.registerDefaultFunctions()`, a **`private mutating func` inside the `LassoParser` target itself**.
- **(b) Native-type-with-methods**, via `LassoNativeTypeRegistry`/`LassoNativeType` (`NativeTypes.swift:1-70`) ‚Äî a type name resolves to `.object(LassoObjectInstance(typeName:))`, and `->method(...)` calls dispatch through `LassoNativeMethod` closures registered on that type. Exactly four built-in types exist today: `web_request`, `web_response`, `date`, `bytes` ‚Äî all registered by `registerDefaultTypes()`, also inside `LassoParser`.
- **(c) Hardcoded block-tag `switch` in `Renderer.swift`** ‚Äî a fixed, closed set of block-tag names, not extensible without editing the renderer itself. Not relevant to this plan.

**The `email_send` stub, verbatim** (`LassoParser/Runtime.swift:918-930`):
```swift
// [Email_Send] (Lasso 8.5 Language Guide, "Process Tags"): a
// process tag -- "does not return a value" per its own doc -- that
// queues/sends a real email via SMTP (-Host/-To/-From/-Subject/
// -Body etc.). This project has no resurrected SMTP client, so a
// real send isn't possible; ...
register("email_send") { _, _ in .void }
```
The accompanying comment is now stale. A matching regression test, `emailSendIsARegisteredNoOpNotAnUnknownFunction` (`LassoParserTests.swift:5622-5634`), documents the real corpus call shape: `email_send: -to='a@example.com', -from='b@example.com', -subject='s', -body='b'` ‚Äî Lasso 8's bare colon-call form, sourced from `importscripts/*.lasso`.

**No sync/async bridge is needed.** The entire render pipeline ‚Äî `Evaluator`, `Renderer`, every `LassoNativeFunction`/`LassoNativeMethod` ‚Äî is `async throws` end to end (confirmed live, and via `main.swift:1172-1183`'s doc comment stating no bridge is needed). `SMTPMailer.send` is itself `async throws` (`PerfectSMTP/SMTPMailer.swift:78-82`), so native closures `await` straight into it once dispatch is solved (¬ß4.0).

**Error propagation ‚Äî two live mechanisms, and `email_smtp`/`email_send` need the second one.** `LassoDynamicQueryExecutor` conformers auto-populate `context.currentError` via `LassoInlineFrame`'s error field ‚Äî specific to the `[inline]` frame machinery, not available to a plain native function/method. `LassoRecoverableError` (`Providers.swift:57-63`) is caught only by `[protect]` (`Renderer.swift:191-212`); anything not caught there becomes a developer error page. **`email_send`/`email_smtp` will throw `LassoRecoverableError`.**

**Dash-param parsing** ‚Äî no new infrastructure needed. `Array<EvaluatedArgument>` already exposes `firstValue(named:)`, `lastValue(named:)`, `strings(named:)`, `lastString(named:)`, `lastInt(named:)`, `hasTruthyFlag(_:)`, `positionalValue(at:)` (`Runtime.swift:1382-1418`).

**Credential/config source precedent** ‚Äî `DatasourceFileConfig` (`LassoPerfectServer/main.swift:472-520`), loaded from a JSON file at `LASSO_DATASOURCE_CONFIG_PATH`, one top-level block per backend, env-var fallback per block. **Correction from review: the "chmod 600" convention this file's own doc comment recommends is documentation-only ‚Äî no code anywhere in `main.swift` actually stats the file or enforces permissions at load time.** Acceptable for a rotatable password; not sufficient on its own for a DKIM private key (¬ß4.6, ¬ß5).

**`FileMakerConnectionRegistry` does not do what this plan's first draft assumed ‚Äî corrected during review, two independent reviews caught this.** It holds **no live connection or pool at all**. Read in full (`Sources/LassoPerfectServer/FileMakerConnectionRegistry.swift`): it's a pure `alias -> (host, port)` *resolution table*, built solely for the admin console's live "switch datasource" feature (`resolve(alias:)`, `switchAlias(_:to:)`). The actual FileMaker connection strategy is confirmed to be the deliberate opposite of pooling ‚Äî `main.swift:833-839`'s own comment states a **fresh `FileMakerServer` per call**, "even though the resurrected `FileMakerServer` is now natively Sendable and could safely be built once and captured," to match pre-resurrection behavior. This plan's proposed shared, pooled `SMTPMailer` (¬ß4.6) is still the right design on its own merits ‚Äî SMTP is a genuinely stateful protocol session in a way FileMaker's HTTP-based Data API isn't, and `SMTPConnectionPool` exists specifically to make pooling safe and worthwhile ‚Äî but it is **not** precedented by `FileMakerConnectionRegistry`, and the plan should not claim it is.

**Perfect-SMTP's actual public surface, verified `public`, including one significant correction:** `SMTPMailer` (`send` single/array/streaming, `composeAndSign`), `EmailMessage`/`EmailAddress`/`Attachment`/`InlineResource` (`Data`-based, no file-loading), `RelayTransport`/`LocalMTATransport`/`DirectMXTransport` and their `Config` types, `DirectMXConfig.allowPrivateAddresses` (defaults `false`, SSRF filter ‚Äî **confirmed this filtering exists only on `DirectMXTransport`; `RelayConfig`/`RelayTransport` has no equivalent field at all**, see ¬ß5), `DNSResolver`, `MTASTSPolicy`/`MTASTSPolicyManager`, `DKIMSigner`/`SigningKey`. **Correction from review: `SMTPConnection` (`public final class`, `SMTPConnection.swift`), `SMTPBootstrap.connect` (`public static func`, `SMTPBootstrapHandler.swift:54`), and `SMTPConnectionPool.withConnection` (`public func`, `SMTPConnectionPool.swift:224`) are ALL already public today** ‚Äî the original draft's claim that `email_smtp`'s live-session design "requires new public surface on Perfect-SMTP" was wrong. See ¬ß4.8.

**Address-list parsing ‚Äî genuinely absent everywhere,** confirmed unchanged from the first draft: neither repo has an RFC-5322-aware, quote-comma-aware splitter. Has to be written from scratch (¬ß4.2).

## 3. Lasso 8.5/9 email API surface (from Perfect-SMTP plan ¬ß6, restated here as the authoritative source for this plan)

Both Lasso versions ship a first-class, near-identically-named built-in email API ‚Äî `[Email_Send]` (8.5, Language Guide Ch. 47) and `email_send` (9, now a method, lassoguide.com) ‚Äî not an external binding, and Lasso 9 kept essentially every 8.5 parameter name verbatim (lowercased). There is no `Send_Mail` tag in 8.5.

**Full shared parameter surface:** `-to`/`-from`/`-subject` (from+subject always required, one of to/cc/bcc required), `-body`/`-html` (one required, both ‚Üí multipart/alternative), `-cc`/`-bcc`, `-htmlImages` (array of paths OR array of `name=data` pairs), `-attachments` (array of paths OR `name=data` pairs, **8MB total-size ceiling documented by both versions**), `-tokens`/`-merge` (mail merge), `-priority` (`High`/`Low`, default `Medium`), `-replyTo`, `-sender`, `-contentType`/`-transferEncoding`/`-characterSet`, `-extraMIMEHeaders`, `-immediate`, `-host`/`-port` (default 25)/`-username`/`-password`/`-timeout`.

**Lasso-9-only additions:** `-ssl`, `-date` (schedule a future send), `-simpleform` (send with no body), and on `email_smtp`: `-clientIp` (HELO/EHLO identity), `-multi`.

**The one 8.5‚Üí9 incompatibility:** 8.5's `-ContentDisposition` lives directly on `[Email_Send]`; Lasso 9 moved it to `email_compose` only.

**Companion surface:** `email_compose` (build-then-send, two-phase), `email_smtp` (low-level connection type ‚Äî `->open`/`->command(-send,-expect,-multi,-read)`/`->send`/`->close`), `email_mxlookup(domain, -refresh, -hostname)`, `email_result`/`email_status` (async job ID + polling).

## 4. Target architecture

### 4.0 The dispatch-registration seam ‚Äî found missing during review, must be designed before any other phase can start

**This section did not exist in the first draft. It is the single most important correction from the review pass, and it changes the shape of every other section below.**

The original draft instructed "replace `Runtime.swift:918-930`'s no-op registration with a real implementation" as if that were a simple in-place edit. It is not. That registration lives inside `LassoNativeRegistry.registerDefaultFunctions()`, a `private mutating func` **inside the `LassoParser` target**, whose `Package.swift` dependencies are `Crypto` only ‚Äî zero awareness of any resurrected library, by design, matching the same "zero knowledge" boundary Perfect-SMTP itself follows in the other direction. If that closure were rewritten to call `SMTPMailer.send` directly, `LassoParser` would need to `import PerfectSMTP`, which breaks the layering this entire ecosystem is built on (a core interpreter target must never depend on a specific resurrected library ‚Äî `LassoPerfectFileMaker`/`LassoPerfectCRUD`/`LassoPerfectSession` all exist specifically so `LassoParser` never has to).

The existing three adapters never actually solve "add new dispatch surface from outside `LassoParser`" ‚Äî they all reuse a **pre-existing, protocol-typed context slot** already designed into `LassoContext` (`inlineProvider`, `sessionProvider`, `requestProvider`; confirmed `public var` at `Runtime.swift:1075-1079`), populated per-request from `main.swift`. `LassoContext.natives`/`.nativeTypes` are technically `public var` and could in principle be swapped out via `LassoContext.init(natives:nativeTypes:...)`, but **no code anywhere does this today** ‚Äî it is an unused, undesigned integration seam, not an established pattern.

**Required new design, split across both repos:**

1. **In `LassoParser`**: define a new, generically-named (Lasso-shaped, not SMTP-shaped) protocol on `LassoContext`, following the exact shape of `inlineProvider`/`sessionProvider` ‚Äî e.g. `public protocol LassoEmailProvider: Sendable { func send(_ arguments: [EvaluatedArgument]) async throws -> LassoValue; /* ...compose, mxLookup, ... */ }` with a new `public var emailProvider: (any LassoEmailProvider)?` slot on `LassoContext`. Rewrite the existing `email_send` (and new `email_compose`/`email_mxlookup`) registrations in `registerDefaultFunctions()` to check `context.emailProvider` and either delegate to it or fall through to today's no-op/error behavior when it's `nil` (mirroring how `"inline"` behaves when `inlineProvider` is unset ‚Äî throws `LassoRuntimeError.inlineNotConfigured` rather than silently no-op'ing, per `Renderer.swift`'s existing pattern; the new email functions should do the same rather than reverting to the current silent `.void` no-op once this exists).
2. **In `LassoPerfectSMTP`**: a concrete `LassoEmailProvider` conformer wrapping `SMTPMailer`, wired into `LassoContext` from `main.swift` exactly like `inlineProvider` is today.
3. **`email_smtp` is a harder, separate problem**, because it needs mechanism (b) ‚Äî a native *type* ‚Äî and **no side-channel-slot equivalent exists for native types at all**. `LassoNativeTypeRegistry.register(_:)` only accepts types built entirely inside `LassoParser` today; there is no `LassoContext`-level slot analogous to `inlineProvider` that a native type's methods can delegate through. This needs its own design, likely a second, type-specific protocol slot (`emailSMTPProvider` or similar) that `email_smtp`'s methods (also newly added to `registerDefaultTypes()` in `LassoParser`, gated behind that slot the same way) delegate to ‚Äî do not assume this is a smaller version of problem 1 above; scope it explicitly once problem 1 is solved and reviewed, since it may surface its own complications.

This is real, reviewable design work in `LassoParser` itself, not just new files in a new target ‚Äî it should be scoped, implemented, and milestone-reviewed as its own deliverable at the start of Phase A (¬ß8), before `email_send`'s actual parameter mapping is implemented, since nothing else in this plan can be exercised end-to-end until it exists.

### 4.3b `email_compose` ‚Äî corrected during Phase C research; the first draft's mechanism-(a) assumption was wrong

**The first draft (¬ß4.0 point 1) assumed `email_compose` could dispatch through the same simple `LassoEmailProvider` free-function slot as `email_send`/`email_mxlookup`. Verified false against real docs.** `email_compose` is a genuine native-type object (lassoguide.com, "Sending Email" ‚Üí "Compose an Email Message"/"type email_compose"), not a free function ‚Äî its own canonical worked example:
```lasso
local(message) = email_compose(-to=..., -from=..., -subject=..., -body=...)
#message->addAttachment(-path='ExampleFile.txt')
[#message->asString->encodeHtml]
```
`->addAttachment`/`->addHTMLPart`/`->addTextPart`/`->addPart` genuinely mutate the object after construction ‚Äî this is exactly ¬ß4.0 point 3's "no side-channel-slot equivalent exists for native types" problem, previously scoped only to `email_smtp`/Phase D. Confirmed real, not a corpus-unconfirmed edge case: it's the reference doc's own primary example.

**Resolution (explicit sign-off given for Phase C, scoped-subset approach):**
- `email_compose` becomes a real `LassoNativeType` (`LassoNativeTypeRegistry`, same mechanism as `date`/`bytes`/`web_request`/`web_response`) in this phase ‚Äî the actual type-registration and method-dispatch machinery is genuine, working code, not stubbed.
- **Constructor** accepts the same dash-params as `email_send` (confirmed: "accepting similar parameters as email_send" ‚Äî full signature: `-to/-from/-cc/-bcc/-subject/-sender/-replyTo/-body/-html/-date/-contentType/-characterSet/-transferEncoding/-contentDisposition/-headerType/-extraMIMEHeaders/-attachments/-attachment/-htmlImages/-parts`; notably NO `-host/-port/-username/-password/-immediate/-tokens/-merge` ‚Äî those are `email_send`/`email_queue`-only). Reuses `LassoSMTPMessageBuilder.build` directly for the parameter parsing + `LassoSMTPAttachmentLoader` for attachment resolution ‚Äî same validated pipeline `email_send` already uses, just stopping short of relay resolution/sending and calling `SMTPMailer.composeAndSign` instead (no envelope, no transport).
- **Two real construction modes, per lassoguide.com**: "If the `-to`, `-from`, and `-subject` parameters are not specified then a MIME part is created, otherwise a full MIME email is created." This phase supports only the full-message mode (all three present) ‚Äî the bare-MIME-part mode's only documented purpose is feeding into another object's `->addPart` (itself deferred below), so it has no reachable use yet. Attempting part-mode construction (any of `-to`/`-from`/`-subject` absent) throws a clear `notYetSupported` error rather than silently misbehaving or guessing at a bespoke single-part composer.
- **Working accessor methods** (non-mutating, real corpus-documented): `->data(-prefix::boolean=?, -force::boolean=?)` (returns the composed MIME text ‚Äî exact `-prefix`/`-force` semantics unconfirmed against either doc source as of this writing, flagged for implementation-time verification; a conservative default if unconfirmable is for both to be no-ops on the base `composeAndSign` output), `->from()`, `->recipients()` (the doc's own `email_queue(-data=#message->data, -from=#message->from, -recipients=#message->recipients)` pattern is the reason these three specifically must work end-to-end, even though `email_queue` itself is out of this plan's scope ‚Äî a real page could still reasonably call these three and use the results itself). `->asString` (used in the guide's own worked example, alongside but not identical to `->data` in the reference doc's method list ‚Äî implementation should verify whether it's a distinct generic object-conversion method or an alias for `->data`, and pick one deliberately rather than leaving one of the two unimplemented by accident).
- **Deferred, not silently ignored**: `->addAttachment`/`->addHTMLPart`/`->addTextPart`/`->addPart` all throw `LassoSMTPFailureKind.notYetSupported` (or equivalent) rather than being unreachable/crashing or silently no-opping. Real until Phase D resolves the shared in-place-mutation-vs-`Evaluator.selfMutatingMethods` design (¬ß4.8 point 2) for `email_smtp`'s `->open`/`->command`/`->close` ‚Äî at that point both types should adopt whichever mutation mechanism is chosen, once, not twice.
- `-attachment` (singular) and `-parts` constructor params: real, present in the type's documented signature, but with unconfirmed real-corpus shape/usage. Implementation should verify against lassoguide.com/the local Language Guide during Phase C; if the shape can't be confirmed with reasonable confidence, treat as `notYetSupported` (consistent with this plan's established "explicit error over guessed behavior" discipline) rather than guess.

### 4.4 `email_mxlookup`

A free function wrapping `DNSResolver.resolveMX(domain:)` ‚Äî genuinely fits the simple `LassoEmailProvider` mechanism-(a) dispatch (no native-type problem here, confirmed).

**Corrected during Phase C research: `-refresh` is NOT a no-op.** lassoguide.com states real Lasso caches MX lookup results across calls, keyed by domain, and only `-refresh` forces a fresh lookup: "The first time an MX record is looked up its result is cached and the same information will be returned on subsequent lookups." `DNSResolver` itself has no caching at all (confirmed ‚Äî it's a stateless `Sendable` struct that dials a fresh query every call, by design, per its own doc comment). This means Phase C needs a small new caching layer in `LassoPerfectSMTP` (e.g. an actor holding `[String: (records: [DNSResolver.MXRecord], cachedAt: Date)]` or similar) sitting in front of `DNSResolver.resolveMX(domain:)` ‚Äî not just a bare pass-through wrapper as the first draft assumed. No TTL/expiry semantics are documented by either source; a reasonable, explicitly-flagged-as-a-judgment-call choice is process-lifetime caching (matches "the same information will be returned on subsequent lookups" literally) unless corpus evidence surfaces a real need for expiry.

**Return shape has a real, worth-flagging inconsistency between the reference doc's prose and its own worked example.** Prose: "a map that describes the MX server for the domain. The map includes the 'domain', 'host', 'username', 'password', 'timeout', and 'SSL' preference for the MX server." Worked example's actual output: `map(domain = gmail.com, host = gmail-smtp-in.l.google.com, priority = 5)` ‚Äî three keys (`domain`/`host`/`priority`), not six, and `priority` isn't even in the prose's list. Implementation should match the worked example's actual shape (`domain`/`host`/`priority`) as the higher-confidence source (a literal shown output beats a possibly-stale prose description), with `host` sourced from the lowest-preference (most-preferred) `MXRecord.exchange` DNSResolver returns and `priority` from that same record's `preference` field ‚Äî and should flag in a doc comment that the prose's extra keys were investigated and deliberately not included, rather than silently dropped.

`-hostname`'s exact semantics still need re-confirming against lassoguide.com during implementation ‚Äî neither this plan nor its reviews, nor the Phase C research pass, found an unambiguous definition beyond the bare signature `email_mxlookup(domain, -refresh=?, -hostname=?)`.

### 4.5 Attachments / inline images ‚Äî corrected per security review to reuse existing code and close two gaps the first draft missed

`-attachments`/`-htmlImages`' `name=data` variant maps directly onto `Attachment`/`InlineResource` (already `Data`-based) ‚Äî and its filename value is already protected by `MIMEComposer.sanitizedFilename`, which strips control characters and neutralizes path-traversal sequences before the value ever reaches a `Content-Disposition` header; the adapter needs no new sanitization for this variant, only for the path-based one below.

The path-based variant needs new adapter-side file-loading code, with:
- **Reuse `LassoSiteServer`'s existing containment helper, don't reinvent it.** `main.swift:1075-1118`'s `isWithinRoot` (`siteRoot.appendingPathComponent(relativePath).standardizedFileURL.resolvingSymlinksInPath()`, then a prefix check against the root) is already symlink-aware and already used for exactly this class of problem elsewhere in this codebase. `LassoSMTPAttachmentLoader` should call it (or a shared extraction of it), not write a second, subtly-different implementation.

### 4.6 Shared mailer lifetime and relay configuration ‚Äî corrected per both concurrency and security review

A `LassoSMTPMailerRegistry` actor. **The first draft's claim that this "mirrors `FileMakerConnectionRegistry`'s role" is dropped ‚Äî see ¬ß2's correction; that type does something orthogonal (live alias re-pointing, no pooling) and FileMaker's actual pattern is deliberately unpooled.** This design stands on its own justification instead: SMTP is a stateful protocol session Perfect-SMTP already built explicit, reviewed pooling machinery for (`SMTPConnectionPool`), and a long-lived, actor-held `SMTPMailer` over `RelayTransport` is the only way to actually realize that benefit across many `email_send` calls in one server process rather than constructing a fresh, unpooled transport per call.

plus `LASSO_SMTP_*` env-var fallbacks for the single-relay smoke-test path, matching the `filemaker`/`mysql` blocks. `-host`/`-port`/`-username`/`-password` (¬ß4.3) no longer select an arbitrary literal host ‚Äî see ¬ß5.

### 4.7 `email_result`/`email_status` ‚Äî net-new job tracking, with explicit open questions from review

Perfect-SMTP's own plan deliberately never built an `enqueue`/`status(of:)` API. `LassoEmailJobTracker` (actor) needs a job-ID scheme (Foundation `UUID()` ‚Äî CSPRNG-backed, 122 bits of entropy; explicitly *not* a sequential/incrementing scheme, stated here so a future "optimization" doesn't quietly weaken it), an in-memory `[JobID: JobState]` store (in-memory-only, matching `DirectMXRetryQueue`'s/`MTASTSPolicyManager`'s identical documented scope boundary in Perfect-SMTP), and a bridge from `-immediate=false`/`-date`-scheduled sends into background `Task`s recording `DeliveryResult`s against the job ID.

**Single-tenant confirmation, new from security review.** `lasso-perfect-server` runs one `LassoSiteServer` per process (confirmed, `main.swift:1798`, the only construction site). There is no cross-tenant isolation boundary for this job store to violate today ‚Äî stated explicitly here so it's a tracked assumption, not a silently-reopened gap if this codebase ever grows a multi-site-per-process mode.

**Explicit unanswered questions for Phase E's implementation, new from concurrency review, given `DirectMXRetryQueue`'s own history of a real background-task-leak bug (a nudge-triggered restart that cancelled a sleep but never actually stopped the superseded task's own loop) caught during its milestone review:**
- What happens to a job's background `Task` if nobody ever polls its status again ‚Äî does it still run to completion and sit in the store (bounded only if there's an eviction/TTL policy, which this sketch doesn't yet have), or is there a cancellation path?
- Does the tracker have any shutdown/drain story for a server restart while a `-date`-scheduled send is still pending, or does it vanish silently (acceptable if documented the same explicit way `DirectMXRetryQueue` documents its own in-memory-only limitation, not left implicit)?
- Confirm the background `Task` for a `-immediate=false` send is deliberately *not* scoped to the originating request's own async context (it must outlive the request), rather than assuming an ordinary `Task {}` created inside a request handler survives past that request by default.

### 4.7b `email_result`/`email_status` ‚Äî resolved during Phase E research; real corpus semantics, and answers to ¬ß4.7's three open questions

**Real signatures, confirmed against lassoguide.com's "Email Sending Status" section (verbatim, not paraphrased):**
```
email_result()                       -- takes NO arguments
email_status(id)                     -- id is email_result's return value
```
`email_result()` is "Can be called immediately after calling `email_send` to get a unique ID string for the queued message" ‚Äî it takes **no parameter at all**; it implicitly refers to whatever `email_send` call most recently completed in the current execution. `email_status(id)` "returns the status of the queued message: 'sent', 'queued', or 'error'" ‚Äî exactly these three lowercase strings, nothing richer.

**The argument-less `email_result()` needs a new per-execution "last job ID" slot, following `LassoContext`'s existing `currentError`-style precedent, not a `LassoContext` mutation from inside the provider (which is impossible ‚Äî see below).** `LassoContext` is a value type threaded through the evaluator via `inout` at the free-function-registration layer (`Runtime.swift`'s `register("email_send") { arguments, context in ... }` closures, matching `LassoNativeFunction`'s typealias), but `LassoEmailProvider.send(_:context:)`'s own signature takes `context: LassoContext` (by value, NOT `inout`) ‚Äî so the provider's implementation (`LassoPerfectSMTP`) has no way to mutate the caller's context directly; any job-ID information has to travel back through `send`'s *return value*, then get stashed into a new `LassoContext` field (e.g. `lastEmailJobID: String?`) by the free-function wrapper in `Runtime.swift`, which does have `inout` access. Concretely: `LassoEmailProvider.send`'s return type needs to widen from bare `LassoValue` to something that also carries the job ID (e.g. a small `Sendable` result struct, or a tuple) ‚Äî a source-breaking protocol change, but this project has already precedented widening this exact protocol every phase (Phase C added `compose`/`mxLookup`, Phase D added the `email_smtp` quartet); the one real conformer (`LassoEmailProviderImpl`) and the one test-double conformer (`LassoParserTests.swift`) both need the same, now-familiar update.

**Job-ID scoping for the failure case.** A job ID should only ever be recorded once an actual send has genuinely been *attempted* (relay resolved, `SMTPMailer.send`/the background-dispatch equivalent invoked) ‚Äî matching real Lasso's own framing ("stored in an outgoing email queue"; you need a validly-parseable, queueable message to get an entry at all). Pre-send validation failures (`LassoSMTPMessageBuilder.build` throwing on a missing `-from`, an unresolvable attachment, etc.) should keep failing exactly as they do today ‚Äî immediate throw, no job ID, unaffected by this phase. Only once a real send attempt has genuinely started does a job exist to track; its outcome (success or delivery failure) is what populates the job's terminal state.

**Where "queued" actually becomes observable.** Every `email_send` call to date (Phases A-D) completes synchronously before returning ‚Äî by the time a caller could call `email_result()`, the send has already resolved to success or failure, so a job's state would only ever be observed as `sent`/`error`, never `queued`. This is exactly what `-immediate=false` and `-date` unlock, and why they were deferred here rather than bolted onto Phase A: `-immediate=false` records a job as `queued` and returns immediately, with a background `Task` performing the real send and updating the job to `sent`/`error` once it completes; `-date=<value>` does the same but the background `Task` first sleeps until the given time before attempting the send (parse `-date` via the existing `LassoDateParsing`/`date` native-type machinery already in this codebase ‚Äî reuse it, don't reinvent date parsing). `-date` implies deferred sending regardless of `-immediate`'s own value (a future date inherently means "not now"); `-immediate=false` alone (no `-date`) means "send as soon as possible, but don't block the request."

**Deliberate divergence from real Lasso's own default (Phase E milestone review, protocol/SMTP pass ‚Äî accepted, not something to fix here).** This codebase's default `email_send` behavior (no `-immediate` given at all) is fully SYNCHRONOUS ‚Äî the call blocks until the real send resolves, exactly like every phase before this one. Real Lasso's own actual default is the opposite: queued/async, with `-immediate=true` being what real Lasso uses to *skip* the queue and send synchronously instead (confirmed against lassoguide.com, an archived lassosoft.com reference mirror, and the local Lasso 8.5 Language Guide PDF ‚Äî all three agree on this direction). The framing above (¬ß4.7b's own "Where 'queued' actually becomes observable" paragraph) previously understated the scope of this divergence, focusing only on the narrower observation that `email_status` would never observe `"queued"` for an ordinary send under this codebase's synchronous default ‚Äî the fuller, more accurate framing is a genuine **request-latency inversion**: every plain `email_send` call in this codebase blocks the requesting page on the full SMTP round trip (connect/EHLO/AUTH/MAIL FROM/RCPT TO/DATA), where real Lasso's own default returns to the page immediately and performs that same round trip later, off a persistent queue table plus a background delivery daemon. This is a real, already-made Phase A architectural decision (a from-scratch, in-process, no-persistent-queue-table implementation vs. real Lasso's persistent-queue-table-plus-daemon model) ‚Äî not a bug, and explicitly not something Phase E should attempt to correct; recorded here honestly, with its full scope, rather than left implicit.

**Answers to ¬ß4.7's three open questions, resolved explicitly (not left implicit) per that section's own instruction:**
1. **Unpolled jobs**: still run to completion and sit in the store ‚Äî bounded by a new eviction policy this phase adds (not deferred): a TTL (24 hours ‚Äî long enough to match the doc's own "would be checked sometime later" framing) plus a hard entry-count cap as a defensive backstop, matching the exact discipline already established for `LassoMXLookupCache` (Phase C) and `LassoSMTPAttachmentLoader`'s count ceiling (Phase B) ‚Äî every prior unbounded-growth gap in this project was found by review after the fact; this one should ship bounded from the start.
2. **Restart/shutdown**: the tracker's store is in-memory-only and vanishes on restart ‚Äî explicitly documented as such (matching `DirectMXRetryQueue`'s own accepted scope boundary), but any background `Task` performing a deferred/scheduled send (or running the eviction sweep) MUST be genuinely cancelled on "Restart Server," following the exact pattern Phase D's milestone review had to retrofit after finding it missing for the connection-reaper task ‚Äî get this right from the start this time, wired into `LassoAdminDelegate`'s restart action alongside `cwpJanitorTask`/`smtpConnectionReaperTask`/`siteServerTask`, not added in a later fix pass.
3. **Task lifetime**: the background `Task` for a `-immediate=false`/`-date` send must be a genuinely detached, server-lifetime-scoped task (not created in a way that's implicitly cancelled when the originating request's own async context completes) ‚Äî needs an explicit test proving the send still completes and the job's state still updates after the triggering request has fully returned, not just an assumption that `Task { ... }` outlives its creator (which is not guaranteed in every construction).

### 4.8 `email_smtp` ‚Äî substantially corrected per review; the original recommendation was built on an incorrect premise

**Two things review found wrong with the first draft's framing, both correcting the actual design tradeoff, not just a citation:**

1. **"Live-session model requires new public surface on Perfect-SMTP" is factually wrong.** `SMTPConnection` (`public final class`, with `public init`, `write`, `writeLine`, `negotiateCapabilities`, `authenticate`, `sendMessage`, `nextReply`), `SMTPBootstrap.connect(host:port:tls:...) async throws -> NIOAsyncChannel<SMTPReply, SMTPCommand>` (`public static func`), and `SMTPConnectionPool.withConnection` (`public func`) are all already public in Perfect-SMTP today ‚Äî no new product, no new cross-repo work needed to reach them. **The real, narrower obstacle**: `SMTPConnectionPool.checkout`/`.release` are `private`, so a connection can't be held open across multiple independently-dispatched `->open`/`->command`/`->close` calls *via the pool's public API* ‚Äî only within one Swift closure passed to `withConnection`. This doesn't block the live-session model at all; it just means `email_smtp` would dial directly via the already-public `SMTPBootstrap.connect` + `SMTPConnection.init`, **bypassing `SMTPConnectionPool` entirely for this one code path** (trading away pooling for this specific raw-command use case, a reasonable tradeoff for a feature whose real corpus usage is unconfirmed anyway).

2. **"Stateless-envelope model, closer to `date`/`bytes`" overstates the fit and elides a real, already-encountered problem in this exact codebase.** `date`'s methods were *originally* mutating and caused a real aliasing bug ‚Äî `LassoValue.object` copies the enum case but not the underlying `LassoObjectInstance` class, so two variables could silently share mutations; the fix (`NativeTypes.swift`'s `date->add`/`->subtract` doc comment) was to make `date` non-mutating and rely on `Evaluator.selfMutatingMethods` (`Evaluator.swift:912-927`, a small hardcoded method-name set) for write-back-on-bare-statement instead. `bytes`' methods (`decodeBase64`/`encodeBase64`/`encodeUrl`) never mutate the receiver at all. **`email_smtp`'s `->open` then repeated `->command` then `->close` sequence needs exactly the cross-call state accumulation this codebase already tried and moved away from for `date`.** Two real, named options ‚Äî the plan should pick one explicitly during Phase D design, not describe a third option that doesn't actually match either existing type:
   - **(a) Genuine in-place mutation** via `LassoObjectInstance.set(_:for:)` (already thread-safe, lock-guarded) inside the native-method closures ‚Äî technically works with zero `Evaluator.swift` changes, but reintroduces the aliasing semantics the `date` migration explicitly moved away from as wrong (though a live connection handle arguably *should* have reference/aliasing semantics, unlike a value-like date ‚Äî this is a real judgment call, not obviously wrong to choose).
   - **(b) Non-mutating-plus-write-back**, following `date`'s actual current pattern ‚Äî add `"open"`/`"command"` etc. to `Evaluator.selfMutatingMethods` and thread the accumulated state through return values. Requires editing core `Evaluator.swift`, not just the new target ‚Äî same category of cross-target work as ¬ß4.0.

**`-clientIp` (Lasso 9's `email_smtp` HELO/EHLO identity override, named in ¬ß3) was entirely unaddressed in the first draft ‚Äî added here.** Maps to `RelayConfig.ehloHostname`, analogous to how `-host`/`-port` can override the configured relay per-call ‚Äî resolve alongside that design (¬ß4.3/¬ß5) rather than as an afterthought.

**Deferral rationale, strengthened per concurrency review with a second, independent justification beyond the (now-corrected) API-surface argument.** SMTP is strictly sequential ‚Äî command/response pairing depends on exactly one party writing and reading a connection in lock-step, an invariant Perfect-SMTP's own `SMTPConnectionPool` enforces structurally via checkout/release exclusivity. A live-session model keyed by a bare opaque session-ID string, with no equivalent checkout discipline, has no structural defense against two concurrent contexts referencing the same session ID and interleaving commands against one connection ‚Äî corrupting the conversation, not just racing harmlessly. This is a real reason to prefer the narrower, already-buildable option independent of the (corrected, now-smaller) implementation-cost argument.

**Recommendation, carried forward with corrected justification:** ship option (a) from problem 2 above (in-place mutation, `date`-migration-aware, explicitly chosen rather than assumed) plus direct un-pooled dialing per problem 1, as the "live-enough" `email_smtp` for Phase D ‚Äî closer to genuine `->command` fidelity than the first draft's stateless-envelope idea, and now correctly understood not to require any Perfect-SMTP-side work. Real corpus usage of `email_smtp`, if any exists, should still inform how much of `->command(-send,-expect,-multi,-read)`'s full surface is worth building versus a narrower subset.

### 4.8b `email_smtp` ‚Äî refined during Phase D research; a third option resolves the mutation-mechanism question better than (a) or (b)

**The real corpus worked example (lassoguide.com, "Communicate with an SMTP Server") resolves several open questions and changes the design in one important way:**
```lasso
local(message) = email_compose(-to=..., -from=..., -subject=..., -body=...)
local(smtp) = email_smtp
#smtp->open(-host='smtp.example.com', -port=25, -username='SMTPUSER', -password='mysecretpassword', -timeout=60)
#smtp->send(-from=#message->from, -recipients=#message->recipients, -message=#message->data + '\r\n')
#smtp->close
```

**Construction is bare (`email_smtp`, no parens, no args) and never dials.** Verified against this codebase's own existing bare-identifier-evaluation code (`Evaluator.swift:19-41`, the `.identifier(name)` case, already built for the `session` native type precedent): when `name` matches a registered `LassoNativeType`, a bare identifier reference constructs `.object(LassoObjectInstance(typeName: name))` directly ‚Äî no arguments, no native-function/provider dispatch at all. This means **the constructor needs zero I/O and zero `LassoEmailProvider` involvement** ‚Äî it's exactly the `date`/`bytes` shape (pure `LassoParser`-side, synchronous), just storing whatever `-host`/`-port`/`-timeout`/`-username`/`-password`/`-ssl`/`-clientIp` values were given (if any ‚Äî `email_smtp(-host=..., ...)` with args is also documented as legal, "can optionally pass in the SMTP server parameters," reached via the ordinary free-function/`.call` path, mirroring `date`'s own registration) as defaults for a later `->open` call that might omit them. All the real work ‚Äî dialing, EHLO, AUTH ‚Äî happens in `->open`, confirmed by the worked example passing every real connection parameter there and never to the bare constructor.

**This resolves the plan's own (a)-vs-(b) dilemma with a better third option: a Swift-side connection registry keyed by object identity, not `LassoObjectInstance` field mutation at all.** `->open`/`->command`/`->send`/`->close` all need to act on the SAME live `SMTPConnection` across independent calls ‚Äî but that live connection (a NIO channel wrapper) has no representable `LassoValue` case at all, so it was never going to live *inside* `LassoObjectInstance`'s `[String: LassoValue]` storage regardless of which mutation mechanism was chosen. The actual design: a new actor in `LassoPerfectSMTP` (e.g. `LassoSMTPConnectionRegistry`) holding `[ObjectIdentifier: SMTPConnection]`, keyed by `ObjectIdentifier(receiver)` ‚Äî `receiver` being the exact `LassoObjectInstance` passed into every registered native method call (`LassoNativeMethod`'s first parameter, `NativeTypes.swift:11-15`). `->open` creates the connection and inserts it keyed by the receiver's identity; `->command`/`->send` look it up by that same identity; `->close` removes it. Since `LassoObjectInstance` is a reference type with real identity (`TypeSystem.swift`'s `== ` is `===`), two Lasso variables both holding the same `email_smtp` object correctly refer to the same live connection ‚Äî the exact reference/aliasing semantics ¬ß4.8 point 2 already noted a connection handle *should* have (unlike `date`'s value semantics) ‚Äî **without needing option (a)'s in-place field mutation OR option (b)'s `Evaluator.selfMutatingMethods`/core-parser edit at all.** No `Evaluator.swift` change is needed for this beyond what Phase C's security fix already added (which is orthogonal ‚Äî it blocks *Lasso-source-level* `object->_field = value` assignment, not a native method's own Swift-side calls into a separate actor; those remain completely unaffected).

**`->open`'s real behavior, per the worked example:** connects (`SMTPBootstrap.connect(host:port:tls:connectTimeout:group:) async throws -> NIOAsyncChannel<SMTPReply, SMTPCommand>`), wraps via `SMTPConnection.init(asyncChannel:ehloHostname:...)`, then performs the full handshake automatically ‚Äî `negotiateCapabilities()` (EHLO), then `authenticate(_:)` if `-username`/`-password` were given (the worked example passes credentials to `->open` and never issues a separate raw `->command` for AUTH) ‚Äî matching "Requires a -host... Also accepts optional -port/-username/-password/-timeout" read as "opens and gets the connection ready to use," not "opens a bare socket." `-host` is required (from either the constructor's stored defaults or `->open`'s own arguments ‚Äî `->open`'s own arguments take precedence when both are given); its absence from both throws a clear error, per the doc's own "Requires a -host" statement.

**`->send(-from::string, -recipients::array, -message::string)` maps directly onto `SMTPConnection.sendMessage(_ envelope: SMTPEnvelope, _ message: SignedMessage) async throws -> [DeliveryResult]`** (already handles the full MAIL FROM/RCPT TO/DATA sequence internally, confirmed by reading its implementation) ‚Äî construct `SMTPEnvelope(mailFrom:recipients:size:)` from `-from`/`-recipients` and `SignedMessage(rfc5322: Array(message.utf8))` from `-message` (the worked example's `#message->data + '\r\n'` is the caller's own responsibility to get right, not something `->send` needs to add on its behalf), call `sendMessage`, and ‚Äî matching this project's established "never let email_send-family calls silently claim success when delivery actually failed" discipline (Phase A's `isFailureOutcome`/`describeOutcome` precedent) ‚Äî throw a clear, catchable error if any `DeliveryResult.Outcome` indicates failure, reusing `LassoEmailProviderImpl`'s existing helpers rather than duplicating that logic.

**`->command(-send=?, -expect=?, -multi=?, -read=?, -timeout=?)`** is genuinely low-level, deliberately unsanitized raw protocol access (`-send`'s whole point is caller-controlled wire content ‚Äî this is not a header-injection surface the way `-extraMIMEHeaders` is, since there's no higher-level abstraction here to protect) ‚Äî maps onto `SMTPConnection.writeLine(_:)` + `.nextReply()`. `-expect` compares against `reply.code`, returning `true`/`false`. `-read`, if truthy, returns the reply text instead of the boolean (exact real-Lasso return shape when both `-expect` and `-read` are given together is not confirmed by any doc source checked so far ‚Äî a real implementation judgment call, flag and document it rather than guessing silently). `-multi`'s real meaning is likely already subsumed by `SMTPReply.lines: [String]` already assembling every continuation line (`250-`) into one reply before `nextReply()` returns it ‚Äî verify this against `DNSWireFormat`/the response decoder directly rather than assuming, and document whichever conclusion is reached.

**Connection lifecycle / leak safety net.** The worked example always calls `->close` explicitly, and this is documented, low-level, rarely-used API ("not generally necessary") ‚Äî but a page that errors out mid-sequence, or simply forgets, would otherwise leak a live connection (and a `LassoSMTPConnectionRegistry` entry) for the life of the process. Add a simple idle-timeout reaper (matching this project's established defensive-timeout culture ‚Äî `SMTPConnectionPool`'s own timeout handling, the CWP session janitor, crawl-report's circuit breakers) ‚Äî e.g. a periodic sweep closing and evicting any registry entry unused for some duration (five minutes, matching `SMTPConnection`'s own default `replyTimeout`, is a reasonable, documented starting point) rather than leaving abandoned connections to accumulate indefinitely.

**Implementation note (Phase D milestone review ‚Äî two findings, both resolved before merge):**

1. **Connection-hijack vector, real security finding, fixed.** Keying `LassoSMTPConnectionRegistry` by bare `ObjectIdentifier(receiver)` with nothing retaining `receiver` meant ARC could deallocate the `LassoObjectInstance` behind a stale key (a page erroring out or returning without `->close` ‚Äî a routine control-flow pattern) while the registry still held a live, already-authenticated `SMTPConnection` under it. If a different, unrelated `email_smtp` object happened to get allocated at that same freed address before the idle reaper swept it, that object's `->command`/`->send` would silently resolve to someone else's connection ‚Äî sending mail through another request's SMTP session without ever authenticating. Fixed by having the registry's dictionary value retain `(receiver, connection)` together, not just `connection` ‚Äî as long as any entry lives, this actor holds a strong reference to its exact original `receiver`, so no other object can ever be allocated at that address and collide with the stale key. See `LassoSMTPConnectionRegistry.swift`'s own doc comment for the full writeup.
2. **SSRF exposure, flagged and gated, not silently shipped.** `email_smtp->open` dials ANY literal caller-given `-host`/`-port` with zero address-routability filtering ‚Äî this is real Lasso's own documented behavior (there is no "named relay" concept for `email_smtp`, unlike `email_send`/`email_compose`'s ¬ß5 design), but it means an unauthenticated/untrusted Lasso template with `email_smtp` available to it can reach internal-network addresses. Real Lasso operators had an actual enforced mitigation for this ‚Äî a per-tag/per-group permission system ("Setup > Global > Tags and Security > Groups > Tags" per the Lasso 8.5 Language Guide) ‚Äî with no equivalent in this codebase. Resolved with an off-by-default operator gate, `smtp.allowEmailSMTP` / `LASSO_SMTP_ALLOW_EMAIL_SMTP`, mirroring this codebase's exact `mysql.allowRawSQL` / `LASSO_MYSQL_ALLOW_RAW_SQL` precedent (`ServerConfig`, `main.swift`) rather than inventing a new pattern. Checked once, at `smtpOpen`'s entry, before any dialing ‚Äî `->command`/`->send`/`->close` need no gate of their own, since they already fail cleanly against a connection that was never opened.

### 4.9 Phase F ‚Äî operator-level policy, resolved during Phase F research

Four independent, separable pieces (¬ß6's Phase F bullet). None require any new `LassoParser`-side dispatch surface ‚Äî all four extend `LassoPerfectSMTP`/`LassoPerfectServer` config and the existing `email_send`/`LassoSMTPMailerRegistry` machinery already built in Phases A/B/E.

#### 4.9a DKIM key config

`SMTPMailer(transport:signer:configuration:logger:)`'s `signer` parameter (`Sources/PerfectSMTP/SMTPMailer.swift:62-72`) is exactly where a `DKIMSigner` (already `MessageSigner`-conforming) plugs in ‚Äî confirmed by reading `SMTPMailer.swift` directly. `LassoSMTPMailerRegistry.init` (currently `Sources/LassoPerfectSMTP/LassoSMTPMailerRegistry.swift:87`) builds one `SMTPMailer` per named relay with `signer` omitted (defaults `nil`); Phase F adds an optional `signer: (any MessageSigner)? = nil` parameter to that initializer, applied uniformly to every relay's `SMTPMailer` it builds ‚Äî one signer, shared across every relay, matching this project's established single-tenant-per-process architecture (¬ß4.7's "no cross-tenant isolation boundary to violate" finding, restated here: there is one operator-controlled sending domain per process, so one signer is the right cardinality, not a per-relay or per-call signer).

**Config shape**, a new `dkim` sub-block inside the existing `smtp` block:
```json
"smtp": {
  "relays": { "...": {} },
  "defaultRelay": "primary",
  "dkim": {
    "domain": "example.com",
    "selector": "s1",
    "keyPath": "/etc/lasso-perfect-server/dkim/s1.private.pem",
    "keyType": "rsa"
  }
}
```
`domain`/`selector`/`keyPath` are required together ‚Äî if exactly one or two (not all three) of `dkim.domain`/`dkim.selector`/`dkim.keyPath` are set, `ServerConfig.load()` throws `ServerConfigError.smtpIncompleteDKIMConfig` at startup (fail fast, matching `missingCWPJanitorAdminAPIConfig`'s established convention), rather than silently signing with a half-configured signer or silently not signing at all. `keyType` is optional, `"rsa"` (default, by far the more common real-world DKIM key type) or `"ed25519"`; any other value throws a clear config error.

**Key loading**, done once at server startup, matching ¬ß4.6's own instruction to "read it once at startup into the long-lived `SMTPMailer`/`DKIMSigner`, never per-call":
- `keyType == "rsa"` (default): read `keyPath`'s contents as a UTF-8 string (PEM text), pass to `SigningKey.rsa(pem:)` (`Sources/PerfectSMTPCore/DKIM/SigningKey.swift:34`, already validates ‚â•2048-bit key size via `DKIMSigner.init`'s own `ConfigurationError.rsaKeyTooSmall` check ‚Äî no new validation needed here).
- `keyType == "ed25519"`: read `keyPath`'s contents as a UTF-8 string, base64-decode it, pass the raw bytes to `SigningKey.ed25519(rawRepresentation:)` (`SigningKey.swift:40`). Ed25519 DKIM keys have no equivalent widely-used PEM convention the way RSA does ‚Äî base64-encoded raw 32-byte seed is this implementation's own documented file format, stated explicitly here rather than guessed at from a corpus example (neither real Lasso doc source mentions DKIM at all ‚Äî confirmed by direct search, zero hits for "DKIM"/"MTA-STS" in both `lassoguide.com/operations/sending-email.html` and the local Lasso 8.5 Language Guide PDF ‚Äî this entire feature is operator/infrastructure policy this codebase adds on top of Lasso's own language surface, not something Lasso's own docs specify a format for).
- `DKIMSigner(domain:selector:signedHeaders:keys:)` constructed with `signedHeaders: []` ‚Äî `DKIMSigner.alwaysOversignedHeaders` (`DKIMSigner.swift:130-133`) already covers `from`/`subject`/`to`/`cc`/`bcc`/`date`/`reply-to`/`sender`/`content-type`/`mime-version`/`list-unsubscribe*` unconditionally, which is every header this codebase's own `EmailMessage` construction ever populates ‚Äî no additional caller-specified header names have a confirmed real use here, so an empty base set is the correct default, not an arbitrary placeholder.
- `canon` left at `DKIMSigner`'s own default (`(.relaxed, .relaxed)`), per ¬ß4.6's original instruction.

**Permission check (¬ß4.6, ¬ß5 ‚Äî the actually-enforced, not just documented, requirement)**: before reading `keyPath`'s contents, `stat` the file (`FileManager.default.attributesOfItem(atPath:)`, reading `.posixPermissions`) and refuse to start ‚Äî `ServerConfigError.dkimKeyFilePermissionsTooPermissive(path:)` ‚Äî if the group or world bits (`0o077`) are set at all. This is a hard startup failure, not a warning, matching ¬ß4.6's explicit instruction that the `DatasourceFileConfig` chmod-600-is-documentation-only precedent is NOT acceptable here (a leaked DKIM private key is not equivalently rotatable to a leaked password ‚Äî see ¬ß4.6's full reasoning, unchanged).

**Where this check lives**: `ServerConfig.load()` (`main.swift`), alongside every other startup-time config validation in this file (`smtpRelayMissingHost`, `smtpInvalidDefaultRelay`, `missingCWPJanitorAdminAPIConfig`) ‚Äî not inside `LassoPerfectSMTP`, which has no file-permission-checking code today and shouldn't gain a narrow one-off for this alone when `main.swift`'s composition root already owns every other "read a config-referenced file, validate it, fail fast" responsibility (e.g. this exact pattern doesn't exist yet for FileMaker/MySQL credentials since those are inline JSON values, not file paths ‚Äî DKIM's `keyPath` is this codebase's first config value that's a path to a *separate* secret file, which is exactly why it needs its own explicit check rather than reusing an existing one).

#### 4.9b MTA-STS enforcement toggle + direct-MX opt-in

**These two are coupled, not independent, confirmed by reading Perfect-SMTP directly**: `DirectMXTransport`'s `mtaSTSPolicyProvider` parameter (`Sources/PerfectSMTP/DirectMX/DirectMXTransport.swift:232,269`) is the ONLY place `MTASTSPolicyManager` plugs in anywhere in Perfect-SMTP ‚Äî `RelayTransport`/`LocalMTATransport` have no equivalent parameter at all, and this makes sense architecturally, not just as an API-surface accident: MTA-STS is a mechanism for the *sending* MTA to decide whether to require TLS when connecting to a *recipient domain's* own resolved MX host, which is meaningless when mail instead flows through a fixed, operator-configured relay (the relay operator, not this codebase, is the one who'd resolve that recipient's MX and would need MTA-STS, if anyone does). So: MTA-STS enforcement has no effect and no meaning unless direct-MX delivery is also enabled. `ServerConfig.load()` throws `ServerConfigError.mtaSTSEnforceRequiresDirectMX` if `smtp.mtaSTSEnforce: true` is set while `smtp.allowDirectMX` is not also `true` ‚Äî a config-shape error caught at startup, not a silently-inert flag.

**Config shape** (extends ¬ß4.6's original JSON sketch, which already showed `allowDirectMX: false` as a placeholder):
```json
"smtp": {
  "allowDirectMX": false,
  "mtaSTSEnforce": false
}
```
Both default `false` ‚Äî matching this codebase's consistent "new network-reaching capability is off by default" posture (`smtpAllowEmailSMTP`, `mysqlAllowRawSQL`).

**Direct-MX opt-in mechanism ‚Äî reuses the existing `-host`-selects-a-relay-name design (¬ß4.3/¬ß5) rather than adding new dash-params or a new selection mechanism.** When `smtp.allowDirectMX` is `true`, `LassoSMTPMailerRegistry` gains one additional entry in its `mailers` map under a **reserved name, `"direct-mx"`**, backed by a `DirectMXTransport`-based `SMTPMailer` instead of a `RelayTransport`-based one ‚Äî selectable via `email_send(..., -host='direct-mx', ...)`, exactly like any operator-configured named relay, with zero new Lasso-facing surface. `ServerConfig.load()` throws `ServerConfigError.smtpReservedRelayName("direct-mx")` if an operator's own `smtp.relays` config happens to use that literal name ‚Äî caught at config-load time, not silently shadowed. `LassoSMTPMailerRegistry.init` gains one more new parameter, `directMX: SMTPMailer? = nil` ‚Äî the fully-constructed direct-MX mailer (DKIM signer already applied, matching ¬ß4.9a's "one signer, every relay including this one" design), built by `main.swift`'s composition root (not by the registry itself), inserted into `mailers["direct-mx"]` at construction time if non-nil.

**`DirectMXTransport` construction (`main.swift`'s `smtp` wiring block)**: reuses the exact same `DNSResolver`/`MultiThreadedEventLoopGroup.singleton` already constructed there today for `email_mxlookup` (`mxResolver`, `main.swift:1149`) ‚Äî one resolver instance serving both features, not a second redundant one. `DirectMXConfig()` left at its library defaults (port 25, `.opportunistic` TLS, standard timeouts) ‚Äî **`allowPrivateAddresses` is never made configurable by this phase, hardcoded to its safe default (`false`)**, a deliberate scope-out: this is the exact SSRF-relevant knob ¬ß5's whole `-host` redesign exists to keep away from any caller-reachable surface, and no real deployment need for internal-address direct-MX delivery has been identified (mirrors this file's own established pattern of not building config surface for an unconfirmed need ‚Äî see `-port`/`-timeout`'s per-call scope-out in `LassoSMTPMessageBuilder.swift`'s doc comment). `mtaSTSPolicyProvider: MTASTSPolicyManager()` (`MTASTSPolicyManager.init(configuration:)`, `Sources/PerfectSMTP/MTASTS/MTASTSPolicyManager.swift:274-282`, defaults are library-sane) passed only when `smtp.mtaSTSEnforce` is `true`; `nil` otherwise, matching `DirectMXTransport`'s own documented "purely additive, opt-in" design for this parameter.

**Idle-connection/retry-queue lifecycle**: `DirectMXTransport` owns its own internal `DirectMXRetryQueue`/`SMTPConnectionPool` (already reviewed, already correct, Perfect-SMTP's own scope) ‚Äî no new reaper/janitor Task is needed on the `LassoPerfectSMTP` side the way `LassoSMTPConnectionRegistry`'s reaper was needed for `email_smtp`'s raw connections (¬ß4.8b); this is a fundamentally different, already-self-managing resource, not a gap to close.

#### 4.9c `-tokens`/`-merge` mail-merge templating (+ `-characterSet`)

**Real semantics, confirmed against lassoguide.com's "Email Merge" section (`https://lassoguide.com/operations/sending-email.html`, fetched 2026-07-20) ‚Äî a genuinely different mechanism than a simple string-substitution pass on one message:**
```
-tokens  -- a map of token names to values, used as the DEFAULT for every recipient
-merge   -- a map of {email address -> {token name -> value}}, one entry per personalized recipient;
            values here OVERRIDE -tokens' defaults for that specific recipient only
```
Worked example (verbatim structure, addresses/names changed for this doc): `-to='john@example.com, jane@example.com'`, `-merge=map('john@example.com'=map('FirstName'='John', ...), 'jane@example.com'=map('FirstName'='Jane', ...))`, `-tokens=map('FirstName'='Lasso User', ...)` ‚Äî sending this produces **two independently-composed messages**, one per recipient, each with that recipient's own token values substituted into the shared `-body` template; "The message to John Doe would contain this text: Dear John Doe,". **This means Phase F's `-tokens`/`-merge` support is a per-recipient personalized batch send, not a single-message content edit** ‚Äî a real, load-bearing correction to how this feature was originally scoped as a simple substitution pass.

**Token markers**: real Lasso's preferred mechanism is the `email_token(name::string)` method (`lassoguide.com`, confirmed), which emits a literal `#TokenName#` marker string into wherever it's called from inside a Lasso-rendered `-body`/`-html`/`-subject` value ‚Äî by the time `LassoSMTPMessageBuilder.build` ever sees that string, `email_token(...)` has already been evaluated by the ordinary render pipeline and returned its marker text, exactly like any other native function call. **This needs one new, trivial free-function registration in `LassoParser`** (`Runtime.swift`, alongside `email_send`'s own registration): `register("email_token") { arguments, _ in .string("#\(name)#") }` (name taken from the sole positional/`-name`-equivalent argument, per the method's documented single-string-parameter signature) ‚Äî pure, synchronous, zero I/O, matching `date`/`bytes`'s existing native-function shape. The doc also states "the `#TOKEN#` marker can be used instead" directly in plain-text bodies that never go through `email_token` at all ‚Äî this codebase's substitution pass (below) must recognize literal `#TokenName#` text regardless of whether it came from `email_token(...)`'s return value or was hand-written, since after rendering there is no way to distinguish the two.

**Scope of substitution ‚Äî reaffirms ¬ß4.3's table row, now implemented rather than deferred**: token substitution applies only to `built.message.subject`/`.textBody`/`.htmlBody` (and, unchanged, address display-names already route through `encodePhrase` independent of this feature) ‚Äî never to `-host` (a relay name, not content), attachment path entries, or `-extraMIMEHeaders` names, exactly as ¬ß5's original scoping decision already required. Implementation: a simple, non-regex literal substring replace of every `"#\(name)#"` occurrence for each `(name, value)` pair in the resolved per-recipient token map ‚Äî a marker with no resolved value in either map is left verbatim, unsubstituted (a conservative, explicit judgment call: real Lasso's own docs don't describe missing-token behavior, and silently deleting an unmatched marker or throwing would both be guesses; leaving it as literal text is the least surprising of the unconfirmed options and is easy to spot in a rendered message if it happens).

**The per-recipient fan-out mechanism ‚Äî reuses `SMTPMailer.send(_ messages: [EmailMessage], envelopeFrom:) async -> [DeliveryResult]` (`SMTPMailer.swift:122`), Perfect-SMTP's own existing bounded-concurrency batch-send API, built and milestone-reviewed specifically for "the common case of sending the same shape of message... to many independent recipients" (that method's own doc comment) ‚Äî exactly this feature's use case, not a new sending path.** When `-tokens` and/or `-merge` is present, `LassoEmailProviderImpl.send` (`Sources/LassoPerfectSMTP/LassoEmailProviderImpl.swift:216`) branches: for each address in `built.message.to`, clone `built.message`, resolve that address's token map (`tokens` map overlaid by `merge[address]` if present ‚Äî matching the doc's own "override... if both are specified" rule), substitute markers into the clone's `subject`/`textBody`/`htmlBody`, and collect the clones into one array passed to the batch `send` overload in a single call (one call, not N sequential single-message sends ‚Äî preserves the connection-pooling benefit `SMTPMailer`'s batch API exists for). The non-merge path (today's existing behavior) is completely unchanged when neither `-tokens` nor `-merge` is given ‚Äî this is a strictly additive branch, not a rewrite of the existing single-message path.

**Explicit scope-out, flagged rather than guessed at**: `-cc`/`-bcc` given together with `-tokens`/`-merge` throws a clear `LassoSMTPFailureKind.invalidParameter` error ("`-cc`/`-bcc` are not supported together with `-tokens`/`-merge` in this phase") rather than guessing at unconfirmed semantics (does every recipient's personalized copy also get the same static cc/bcc list? does cc/bcc get its own, unmerged copy? neither lassoguide.com nor the local Language Guide PDF says) ‚Äî matching this codebase's own stated discipline ("if the shape can't be confirmed with reasonable confidence, treat as `notYetSupported`... rather than guess," `LassoSMTPMessageBuilder.swift`'s own doc comment, applied here to a parameter-combination rather than a single parameter).

**Job-tracking interaction (¬ß4.7/¬ß4.7b) ‚Äî one job ID per `email_send` call, aggregate state across the whole batch, not one job per recipient.** Real Lasso's own `email_result()`/`email_status(id)` docs describe a single ID per `email_send` call with no indication of any multi-ID/batch-tracking surface ‚Äî matching the single-job-per-call design Phase E already built, extended here rather than redesigned: the batch call's combined `[DeliveryResult]` (across every recipient clone) is checked with the exact same `isFailureOutcome`/`describeOutcome` aggregate-failure logic the non-merge path already uses (`send`'s existing `results.first(where: { provider.isFailureOutcome($0.outcome) })` pattern, `LassoEmailProviderImpl.swift:370,425`) ‚Äî the job resolves to `.sent` only if every recipient's clone delivered successfully, `.error(...)` (citing the first failing recipient, matching the existing single-failure-citation convention) if any one clone failed. This is an explicit, flagged judgment call (an aggregate boolean loses which specific recipients failed within a successful-overall-job's job-state string), not a discovered requirement ‚Äî a finer-grained per-recipient job-state representation would be a genuine `LassoEmailJobTracker` schema change with no confirmed real-corpus need driving it now, so it's scoped out here the same way this project scopes out other unconfirmed-demand config surface elsewhere in this plan. `-immediate=false`/`-date` deferred sending (Phase E) composes with this batch path unchanged: the same background-`Task` mechanism performs the batch send instead of a single send, updating the one job ID's terminal state from the batch's aggregate result exactly as described above.

**`-characterSet` ‚Äî folded into this phase as a small, unrelated, zero-ambiguity additive fix, not a mail-merge feature.** Currently in `LassoSMTPMessageBuilder.unsupportedParameterNames` (`LassoSMTPMessageBuilder.swift:208`) alongside `-tokens`/`-merge`, throwing `notYetSupported` ‚Äî but ¬ß4.3's own table row (line 120) already specifies its correct, trivial mapping (`arguments.lastString(named: "characterSet") -> EmailMessage.charset`, defaulting to `EmailMessage`'s own `"utf-8"` default when absent) with no design ambiguity at all; it was simply never wired through when `-contentType`/`-transferEncoding` were implemented in Phase B. Removed from `unsupportedParameterNames` and wired in this phase since it's essentially zero marginal cost once this file is already being touched for `-tokens`/`-merge`, and leaving a fully-specified, zero-risk one-line mapping throwing `notYetSupported` indefinitely serves no purpose.

## 5. Cross-cutting concerns

**SSRF exposure via `email_send`'s `-host` ‚Äî substantially revised, this is now the plan's most important open-decision resolution, per the security review's critical finding.** The first draft's "operator opt-in flag" recommendation does not actually close this gap: `RelayConfig`/`RelayTransport` has **no address-routability filtering at all** ‚Äî that machinery exists exclusively on `DirectMXTransport` via `allowPrivateAddresses`. If a per-call `-host` override were implemented as "spawn a short-lived `RelayTransport` to the caller-supplied host" (as ¬ß4.6's first draft sketched), flipping an opt-in flag would grant the exact internal-network-connect primitive (`127.0.0.1`, RFC 1918 ranges, cloud metadata endpoints, this same server's own FileMaker/MySQL/admin-API ports) that `DirectMXConfig.allowPrivateAddresses` exists to prevent ‚Äî with none of that filtering applied, because the connection never goes through `DirectMXTransport`. A single process-wide boolean is also the wrong shape regardless of the filtering question, since it would apply to every page on the site, not just ones an operator specifically audited for this need.

**Corrected design: `email_send` never accepts an arbitrary literal `-host`.** ¬ß4.6's config now defines named relays (`relays: {"primary": {...}, "marketing": {...}}`); `-host` (when given at all) selects a name from that map, never a literal hostname/IP. `-to`'s domain carries no independent SSRF risk on the default relay path (only `RCPT TO` reaches the fixed, operator-configured relay ‚Äî its own domain's DNS is never resolved by this library). If a genuine literal-host escape hatch is ever needed for a direct-MX-adjacent use case, it must route through `DirectMXTransport` with `allowPrivateAddresses: false`, never `RelayTransport`, stated explicitly rather than left for an implementer to discover after the fact.

**`-tokens`/`-merge` scope ‚Äî see ¬ß4.3's table row.** Perfect-SMTP's own header-injection defenses (`HeaderEncoder.rejectHeaderInjection`, `MIMEComposer`'s `extraHeaders` denylist) apply at compose time regardless of how a string was assembled, so merge-substituted `-body`/`-subject`/display-name content is already covered. The actual gap is scope: merge substitution must never reach `-host`, attachment paths, or `-extraMIMEHeaders` *names* ‚Äî none of those are header *values* Perfect-SMTP's injection defenses cover, and `-host` specifically feeds directly into the SSRF concern above.

**Credential handling.** Follows ¬ß2's `DatasourceFileConfig` precedent for ordinary username/password secrets; DKIM key material needs the stricter, explicit handling in ¬ß4.6.

**DKIM/MTA-STS exposure.** Operator/security-posture decisions, belong in the `smtp` server-config block (¬ß4.6), not per-call dash-params ‚Äî unchanged from the first draft, no correction needed here.

**Testing strategy.** `LassoSMTPAddressList` needs pure unit tests with quote/comma edge cases. `email_send`/`email_compose`/`email_mxlookup` need Lasso-parser-level end-to-end tests (matching `LassoParserTests.swift`'s style) plus ¬ß4.0's new `LassoEmailProvider` slot needs its own tests analogous to how `inlineProvider`/`sessionProvider` are exercised. Reuse Perfect-SMTP's fake-server test patterns rather than a new mocking layer. `LassoSMTPAttachmentLoader` needs adversarial tests (`../` traversal, symlink escape, regular-file-type rejection, TOCTOU-relevant ordering, count+size ceilings). `email_smtp`/`email_result`/`email_status` test strategy depends on the ¬ß4.8/¬ß4.7 decisions above.

## 6. Phasing

- **Phase A ‚Äî Dispatch seam (¬ß4.0) + `email_send` core.** The `LassoEmailProvider` protocol/context-slot design in `LassoParser`, its concrete conformer in `LassoPerfectSMTP`, then `email_send`'s actual parameter mapping (`-to`/`-cc`/`-bcc`/`-from`/`-subject`/`-body`/`-html`/`-replyTo`/`-sender`/`-priority`/`-extraMIMEHeaders`), the named-relay config block (¬ß4.6, minus DKIM), the corrected SSRF-safe `-host` design (¬ß5). **This phase now spans both `LassoParser` and `LassoPerfectSMTP` ‚Äî larger than the first draft's Phase A, since ¬ß4.0 wasn't previously accounted for.** First genuinely usable release, covering the real corpus usage already found.
- **Phase B ‚Äî Attachments/inline images.** Both shapes, the reused `isWithinRoot` containment check, regular-file+TOCTOU handling, size+count ceilings, `-ContentDisposition`. `-contentType`/`-transferEncoding` explicitly scoped out (error) or deferred to Phase F per ¬ß4.3's decision.
- **Phase C ‚Äî `email_compose`, `email_mxlookup`.** **Revised during Phase C research (¬ß4.3b): `email_compose` needs a real native-type registration, not the simple free-function slot the first draft assumed** ‚Äî it's a genuine `LassoNativeType` with the same "no side-channel slot for native types" problem ¬ß4.0 point 3 flagged for `email_smtp`. Scoped subset for this phase (explicit sign-off given): full-message construction (reusing `LassoSMTPMessageBuilder`/`LassoSMTPAttachmentLoader` directly) plus the non-mutating accessors `->data`/`->from`/`->recipients`/`->asString`; the mutating builder methods (`->addAttachment`/`->addHTMLPart`/`->addTextPart`/`->addPart`) throw `notYetSupported` until Phase D resolves the shared mutation-mechanism design once for both `email_compose` and `email_smtp`. `email_mxlookup` needs a new domain-keyed caching layer in front of `DNSResolver.resolveMX` (¬ß4.4) ‚Äî real Lasso caches MX lookups; `DNSResolver` itself doesn't.
- **Phase D ‚Äî `email_smtp`.** ¬ß4.0 problem 3 (the native-type dispatch slot) resolved first, then the corrected design from ¬ß4.8 (direct unpooled dialing + explicit in-place-mutation choice), `-clientIp`.
- **Phase E ‚Äî `email_result`/`email_status`.** The job-tracking layer with ¬ß4.7's now-explicit open questions resolved during implementation, not deferred past it. **Revised during Phase E research (¬ß4.7b): `email_result()` takes no arguments** (implicitly refers to the most recent `email_send` call, via a new `LassoContext.lastEmailJobID` field set by the free-function wrapper, since the provider itself can't mutate the caller's context) ‚Äî `LassoEmailProvider.send`'s return type widens to carry the job ID alongside its existing `LassoValue`. `-immediate=false`/`-date` now genuinely defer sending via a background `Task`, bounded by a TTL+count-capped `LassoEmailJobTracker`, with restart-cancellation wired in from the start (not retrofitted after review, per Phase D's precedent).
- **Phase F ‚Äî Operator-level policy.** DKIM key config (with the stricter loading/permission checks from ¬ß4.6), MTA-STS enforcement toggle, direct-MX opt-in, `-tokens`+`-merge` templating (`-contentType`/`-transferEncoding` already resolved in Phase B ‚Äî see ¬ß7 item 5). **Resolved during Phase F research ‚Äî see ¬ß4.9a/b/c**: DKIM signing plugs into `LassoSMTPMailerRegistry`'s existing per-relay `SMTPMailer` construction via one shared signer; MTA-STS enforcement is coupled to (meaningless without) direct-MX, which itself reuses the existing `-host`-selects-a-relay-name mechanism via a reserved `"direct-mx"` registry entry, no new dash-params; `-tokens`/`-merge` turned out to require a per-recipient personalized batch send (`SMTPMailer`'s existing batch-send API), not a single-message substitution pass ‚Äî a real, load-bearing correction to the original scoping. `-characterSet` folded in as a small, unrelated, already-fully-specified additive fix.

Each phase gets its own branch and its own milestone review (architecture, concurrency, protocol, security) ‚Äî unchanged discipline from the first draft, still correct.

## 7. Open decisions requiring explicit sign-off before implementation starts

1. **`email_smtp`'s design (¬ß4.8, ¬ß4.8b)** ‚Äî **resolved, no longer open, and no longer what this item originally said**: an earlier revision of this item recommended explicit in-place mutation (option (a)) on the native-type object; ¬ß4.8b's own later research superseded that with a better third option, actually implemented ‚Äî direct unpooled dialing via `SMTPBootstrap.connect`/`SMTPConnection`, with a Swift-side connection registry (`LassoSMTPConnectionRegistry`, an actor keyed by object identity) holding the live connection entirely outside `LassoObjectInstance`, no in-place field mutation at all. Phase D's milestone review additionally hardened this design: the registry's map value now retains `(receiver, connection)` together (closing a real connection-hijack vector ‚Äî see ¬ß4.8b's implementation note), and `email_smtp->open` is gated behind an off-by-default `smtp.allowEmailSMTP` flag (mirroring `mysql.allowRawSQL`) given real Lasso's `email_smtp` dials arbitrary literal hosts with no SSRF filtering.
