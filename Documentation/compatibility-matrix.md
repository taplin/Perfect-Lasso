# Lasso Compatibility Matrix

Status values: `M1` parser milestone, `M2` runtime milestone, `DB` database
milestone, `Later`, `Research`, and `Unsupported`.

Corpus evidence is drawn from two internal production Lasso codebases used to
validate this matrix during development (one Lasso 8-heavy e-commerce corpus,
one Lasso 8-heavy multi-application corpus with a smaller Lasso 9 startup
library). Codebase names are intentionally omitted from this document; see
local project notes for the source mapping.

| CRLF (`\r\n`) line endings | N/A | N/A | Real startup folder (all Windows-authored `.inc` files) | M2 implemented, `TemplateScanner.init` — Swift's `Character` treats `\r\n` as one grapheme cluster that matches neither the standalone `\r` nor `\n` most newline checks compared against, so every CRLF-terminated file previously risked `skipLineRemainder()` silently swallowing an entire block body past its real closing brace. Normalized once at the earliest entry point (all downstream parsers operate on already-normalized substrings) |
| `library()` per-process caching (root cause of the `excludeBots`/`_begin.lasso` "`->contains` on a non-`.string`" symptom above) | N/A | Primary | Real corpus — every page calls `library('/_begin.lasso')` by hand; `_begin.lasso` itself calls `library()` on a component it needs from the site's `components/` folder | M2 fixed — root-caused via LassoSoft's own `library_once`/`[Library_Once]` documentation ("if used multiple times referencing the same Lasso page then only the first will actually perform the include"): that dedup is scoped to a single page's own render, not the server process's lifetime. This interpreter's `loadedLibraries` set lived on the shared, process-wide `LassoTagRegistry` (`main.swift`'s own comment said so explicitly: "shared across every request... not re-parsed or re-registered per request"), so any top-level executable code in a `library()`'d file — e.g. `_begin.lasso:66`'s live `excludeBots(web_request->header('USER-AGENT'), ...)` call — only ever ran once, ever, for the whole server process; every later request silently no-opped. Moved `loadedLibraries` onto `LassoContext` (fresh per request) while tag/type *definitions* stay on the shared `tagRegistry` as before. This is the most likely real explanation for the `->contains` symptom: whatever `web_request->header('USER-AGENT')` evaluated to on the one first-ever request got permanently baked in |
| Lasso 8 `condition ? whenTrue | whenFalse` conditional-expression operator | Primary | Compatibility | Real corpus — appears in one of `_begin.lasso`'s own component dependencies and independently across nearly every top-level page, found live-verifying the `library()` fix above once requests started actually reaching this previously-unreached code | M2 implemented, value form only (`ExpressionParser.parseExpression` — recognized only when starting a fresh top-level expression, so it doesn't collide with assignment/binary-operator precedence). The bare statement-guard form with no `|` branch (`condition ? statement`, seen in `Auto_Record.inc`/`mini_cart_tag.inc`) is a separate dialect, not implemented |
| `void` vs `null` — member dispatch on a lookup-miss value (e.g. `action_param('template')->size` when no such param is present) | Yes | Yes | Real corpus — same near-universal `log_critical(...)` line across most pages, found immediately after the ternary fix above once rendering reached this far | M2 implemented — researched Lasso 8/9 documentation directly rather than guessing: `null` sits at the root of Lasso 9's type hierarchy and stays genuinely strict (member-dispatch failure only falls back gracefully if the type defines `_unknowntag`; otherwise "an error will result that may terminate processing" — matching LassoSoft's own 8.5→9 migration notes, which flag null/empty-string coalescing as a deliberate tightening, not a bug). But lookup-miss methods like `web_request->param()`/`action_param()`/header/cookie don't return `null` in real Lasso — LassoGuide's own docs state "if no argument matches, a `void` value is returned." `void` is a distinct built-in type from `null`. This interpreter had every param/header/cookie "not found" fallback returning `.null` (11 call sites across `NativeTypes.swift`/`Runtime.swift`/`main.swift`) and had no member-dispatch case for either `.null` or `.void`. Fixed: those 11 sites now return `.void`; `Evaluator.member` treats `.void` as an empty string for any member access (`->size` → 0, `->contains` → false, etc. — delegates to the existing `.string("")` dispatch, matching how `.void` already behaved for truthiness/output elsewhere in this runtime); `.null` itself stays fully strict, unchanged. Also fixed a related bug the parser had baked in: the literal `void` keyword previously parsed to the same `.null` expression node as `null` (harmless before this distinction existed) — now `void` parses to its own `.void` node |
| `X->contains(...)` on a non-`.string` receiver in the real `excludeBots`/`_begin.lasso` bot-exclusion flow | Yes | Yes | Real corpus, live request flow | Fully resolved and live-verified end to end — see the `library()` per-process caching fix above for the root cause. Live HTTP verification against the real site: normal/bot/normal/bot/normal user-agent sequence against a real top-level page correctly alternates 200/200/200/302/200 (the bot redirect fires exactly on the request with a matching UA, not stuck from an earlier request, not permanently disabled). Corpus sweep across the real site's top-level pages: the large majority now render cleanly end to end (0 did before this investigation); the remaining failures are unrelated (missing include files on disk, a distinct legacy `if` dialect gap, a path-boundary check, one unrelated `unsupportedExpression("")` — none connected to library scoping, the ternary operator, or void/null dispatch) |
| Colon-call block openers (`if:(condition); ... else; ... /if;`) | Primary | Compatibility | Real corpus — a real top-level page's server-name branching used this exclusively instead of the parenthesized-call style | M2 implemented — `ScriptBodyParser.parseBlockOpening()` required the block keyword to be immediately followed by `(`; a colon fell through to being parsed as an ordinary colon-call expression (`if` treated as a bare function name), throwing `unknownFunction("if")`. Fixed by accepting an optional `:` immediately before the `(` |
| Relative `include()`/`library()` path resolution when the *including* file was itself reached via a leading-slash, site-root-relative path | Yes | Yes | Real corpus — a real page's cookie/session setup include chain used exactly this shape | M2 fixed — `LassoFileSystemIncludeLoader.loadInclude` stored `includingPath` verbatim; `URL(fileURLWithPath:relativeTo:)` treats any string starting with `/` as a literal filesystem-absolute path and silently ignores `relativeTo`, so the parent directory resolved against the real filesystem root instead of the site root, and any relative `include()` from inside that file failed `pathOutsideRoot`. Fixed by stripping the leading slash from `includingPath` before use, matching the normalization already applied to `path` |
| `[Cache(-Name=..., -Expires=...)]` output-caching tag | Primary | Compatibility | Real corpus — a real page's template wraps sidebar content in it | M2 implemented as a no-op native — this interpreter has no output-caching layer (every render is already computed fresh), so the opening call does nothing and the wrapped body still renders normally; the matching `[/Cache]` needed no handling of its own, already covered by existing generic legacy-closing-tag support |
| `[noprocess]`/`[no_process]` raw-content escape hatch | Primary | Compatibility | Real corpus — used correctly by ~10 real templates to wrap embedded JavaScript; the one real page found missing it crashed on `[j++]` inside unwrapped `<script>` content, since every `[...]` is scanned as an ordinary Lasso bracket tag regardless of surrounding context, and `++` isn't a valid Lasso operator | M2 implemented in `TemplateScanner` — content between `[noprocess]` and `[/noprocess]` is emitted as verbatim text, never scanned for nested `<?lasso ?>`/`[ ]` constructs at all. Checked against real corpus usage before implementing (confirmed real templates already rely on this exact mechanism) rather than inventing speculative malformed-expression recovery behavior |
| `protect` / `error_currentError` recoverable-error model | Yes | Yes | Lasso 8.5 Language Guide (`References/Lasso/`) — documented `[Inline: -Add, ...][Error_CurrentError: -ErrorCode]: [Error_CurrentError][/Inline]` pattern | M2 implemented, first pass — see `Documentation/error-protect-model-plan.md`. New `LassoErrorState`/`LassoRecoverableError` (`Providers.swift`); `LassoContext.currentError`/`lastError` (request-local, reset to "No Error" per fresh context); `LassoInlineFrame` carries an `error` field (defaulted, so no existing caller breaks) that `pushInlineFrame` propagates into `currentError` automatically. `protect` (`Renderer.swift`) now genuinely catches only `LassoRecoverableError` — `return`/`abort` (the `returnSignal` short-circuit) and fatal `LassoRuntimeError`s pass through untouched, matching the plan's three-way recoverable/control-flow/fatal split. `error_currentError` (message, or code via `-errorCode`) implemented. Deferred to the inline-write-raw-sql pass: real database/connector failures actually constructing frames with non-default error state (currently only reachable via a synthetic test native) and the full Lasso 8.5 error-code/constant list (`error_noError`, `error_addError`, etc. — needs extracting from the now-locally-available `Lasso 8.5 Language Guide.pdf`, not invented from memory) |

## Milestone 1 Acceptance

- Every fixture lexes without losing source text or delimiter information.
- Lasso 8 and Lasso 9 calls normalize to shared expression nodes.
- AST nodes retain source ranges and original syntax dialect.
- Unsupported syntax produces diagnostics and recoverable unknown nodes.
- No fixture contains production credentials or personal data.
